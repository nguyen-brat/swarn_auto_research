RAFT: Reward rAnked FineTuning for Generative Foundation Model Alignment

RAFT is the kind of alignment method that makes sense the moment you say it out loud: ask the model for several answers, keep the one the reward model likes best, and train the model to imitate those winners.

The paper’s larger point is that you do not always need full PPO-style reinforcement learning to get the benefits of preference alignment. If you can generate enough candidate responses and rank them reliably, you can turn alignment into a repeated supervised-learning loop. That shift is the whole trick. The reward model stops being a gradient source and becomes a selector.

The basic idea

The problem RAFT is trying to solve is familiar. Standard supervised fine-tuning is stable, but it only learns from a fixed dataset. PPO-style RLHF can use reward feedback more directly, but it is memory-heavy, sensitive to reward scaling, and much harder to run cleanly.

RAFT sits between those two extremes. It keeps the simplicity of fine-tuning, but it refreshes the training data with the model’s own higher-scoring samples.

The intuition is simple. If the model can already generate a few decent candidates, then the reward model can act like a judge that picks the best one in each small batch. Training on those winners shifts the model toward better behavior without the machinery of on-policy RL.

Why this is different from PPO

The key difference is not just implementation convenience. It is the direction of learning.

In PPO, the reward signal pushes on the policy directly. That is powerful, but it also means the optimizer has to manage rollouts, value estimation, KL control, and multiple models at once. The training loop is tightly coupled.

In RAFT, generation and updating are decoupled. You first produce samples, then rank them, then fine-tune. That separation matters for two reasons:

  • It reduces memory pressure, because you do not need to keep the whole PPO stack alive during policy improvement.
  • It makes the learning objective feel closer to SFT. The model is updated by imitation on selected examples, not by backpropagating through a reward-driven control loop.

That is why the paper describes RAFT as SFT-like. The fine-tuning step is ordinary supervised learning. The novelty is in how the training set is produced.

The loop, one stage at a time

The paper frames RAFT as an iterative process. Each round has three pieces.

First, the current model samples K responses for each prompt. A temperature parameter controls how diverse those samples are. Higher temperature means more variety, which gives the reward model a wider set of candidates to choose from.

Second, the reward model scores those K candidates and keeps the highest-ranked one for each prompt. This is local best-of-K filtering: the comparison happens within a prompt, not across prompts. That matters because prompt-level reward models are usually only meaningful relative to answers to the same input.

Third, the model is fine-tuned on the selected winners. After that update, the model should be slightly better, so the next round of sampling starts from a stronger policy.

The loop is easy to remember because each stage has one job:

  • generate alternatives,
  • select the best alternative,
  • imitate the selected alternative,
  • repeat.

RAFT is basically trying to approximate an ideal policy that would always output the highest-reward answer for each prompt, but it does that through repeated local filtering instead of a global search over all possible outputs.

Why best-of-K is the heart of the method

The best-of-K view is the cleanest way to understand RAFT.

If you sample many responses and then keep the one with the highest reward, you are already using the reward model as a decision rule at inference time. RAFT then turns that same idea into training data.

That matters because the induced best-of-K policy is often stronger than the raw policy. Even if the base model is only okay, the best response among several samples can be noticeably better. RAFT learns from that advantage instead of treating it as a one-off inference trick.

There is a tradeoff, though. Larger K usually helps because it gives the reward model more chances to find a strong response. But the inference cost rises with K, because you have to generate and score more samples. So RAFT is not free: it trades extra sampling cost for a better filtered dataset.

In the paper’s language, the marginal gain from larger K diminishes, which is why the iterative loop matters. You do not want to overcommit to a huge batch size if repeated smaller improvements can get you there with less waste.

Why a KL penalty can still be useful

RAFT does not require an explicit KL penalty, but the paper shows how to add one when you want the updated model to stay closer to the initial policy.

The idea is familiar from RLHF: you want reward improvement, but you also want to avoid drifting so far that the model starts producing odd or brittle outputs. In RAFT, that regularizer is optional because the method is already fairly stable, but it can still help when you care about fluency or want to avoid overfitting to the reward model.

Conceptually, the KL term says: improve the reward, but do not move too far from the original generator. The paper uses the current model and the reference model to rank samples with this adjusted score when KL regularization is enabled.

One useful detail here is that RAFT uses reward ranking rather than absolute reward value. That makes it less sensitive to reward scaling. If the reward model is shifted or rescaled, the ordering can still be useful even when the exact numeric values are not. This is one reason the method is relatively robust compared with PPO, which often depends much more on the scale and calibration of the reward signal.

What the method is really doing to the data

It helps to think of RAFT as a data construction method disguised as an alignment algorithm.

Each training round manufactures a cleaner dataset from the model’s own outputs:

  • the prompt batch is the anchor,
  • the reward model removes low-value samples,
  • the winner set becomes the next fine-tuning set,
  • the updated model becomes the new sampler.

That is a nice mental model because it explains both the strength and the weakness of the method.

The strength is that the model keeps training on responses that are actually close to its own behavior distribution. The data is not artificially far away from what the model can currently produce, so supervised fine-tuning can make useful updates without fighting the model’s current preferences too hard.

The weakness is that everything depends on the quality of the judge. If the reward model is noisy, biased, or easy to exploit, then the filtered dataset can drift toward proxy objectives instead of true alignment. RAFT does not remove reward hacking. It just makes the failure mode easier to observe because you can inspect the selected samples directly.

What the paper shows in practice

On the language-model side, the paper reports that RAFT can beat PPO on reward while keeping perplexity slightly better. The headline result is that the RAFT-aligned LLaMA-7B checkpoint reaches a higher reward than the PPO baseline and does so without the same level of instability.

That is the practical payoff of the sample-filter-finetune recipe: you get a competitive alignment gain without paying the full complexity cost of PPO.

The paper also reports that the model’s diversity metrics are preserved or improved rather than collapsing. That matters because alignment methods often buy reward by narrowing the model’s behavior. RAFT’s results suggest that best-of-K filtering can improve preference scores without forcing the model into a degenerate, repetitive style.

The learning curve is especially informative. As the model improves, the induced best-of-K policy improves too, which in turn gives the next round of training better examples. That creates a positive feedback loop that is still simpler than PPO but more dynamic than static SFT.

The same recipe works beyond text

One reason RAFT is interesting is that it is not tied to autoregressive language models. The same recipe also works for diffusion models.

That is a strong clue about what the method actually needs. It does not need token-level policy gradients. It needs a generator, a reward function, and enough samples to rank.

In the diffusion experiments, RAFT is used to restore and improve image generation quality at a lower cost than DDPO. The paper shows this with a resolution adaptation setup for Stable Diffusion 1.5.

What to notice in that figure is not just that the samples look better. It is that the same reward-ranked fine-tuning loop can recover behavior the base model had largely lost. That is a useful reminder that alignment here is broader than “make chat answers nicer.” It can also mean “push a generative model toward outputs a reward function prefers.”

The paper also uses RAFT for text-image alignment at higher resolution, where the reward is based on OpenCLIP scoring. This is the same high-level idea again: generate several candidates, let the reward function decide which ones are most aligned with the prompt, and fine-tune on the selected set.

What you should take from that image is that RAFT is trying to correct a mismatch between what the prompt asked for and what the generator actually emphasized. The method does that by repeatedly showing the model which samples best satisfy the scoring function.

Why the method can be surprisingly robust

There are a few reasons RAFT is attractive as an alignment recipe.

First, it is easier to tune than PPO because it has fewer moving parts. You still have K, sampling temperature, and maybe a KL coefficient, but you do not have the full actor-critic RL stack.

Second, it is easier to run on limited hardware because generation and fine-tuning are separated. The paper emphasizes that you only need to load one model at a time during each stage.

Third, it is naturally off-policy. You can generate the candidate responses from the current model, but the framework also leaves room for other generators, including stronger teachers. That is why the paper can use RAFT for distillation as well.

Fourth, the method is ranking-based rather than scale-based. If the reward model is imperfect but still mostly orders candidates correctly, RAFT can still make progress because it only needs the relative winner in each prompt group.

That last point is subtle but important. The model does not have to trust the exact numeric reward as a calibrated truth. It only needs the reward model to be good enough to separate better samples from worse ones.

What can go wrong

RAFT is not magic, and the paper is clear about that.

If the reward model is poor, the selected samples can be misleading. If K is too small, the best candidate may not be much better than the rest, so the filtered set does not improve much. If K is too large, inference cost rises quickly. If the reward model is exploitable, the filtered dataset can drift toward reward hacking.

There is also a compute tradeoff in the other direction: RAFT saves memory relative to PPO, but it can still spend more inference time as K grows. So the method is simpler in optimization terms, but it is not free in sampling terms.

That leaves a practical design question the paper does not fully settle: how should you choose K and temperature under a tight compute budget? The answer depends on how much exploration your current model needs and how confident you are in the reward model’s ranking quality.

The easiest way to remember RAFT

Think of RAFT as “best-of-K, then imitate the winner.”

That phrase is almost the whole method:

  1. sample several responses for each prompt,
  2. score them with a reward model,
  3. keep the top one,
  4. fine-tune on those winners,
  5. repeat until the reward stops moving.

If PPO is about continuously pushing on a policy with a learned reward, RAFT is about repeatedly turning reward into a cleaner supervised dataset.

That difference is why the paper matters. It shows that alignment does not always need to look like heavy reinforcement learning. Sometimes the better move is to let the reward model choose the training data, then let supervised learning do the rest.
