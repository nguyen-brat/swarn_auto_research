Absolute Zero Reasoner

Absolute Zero Reasoner, or AZR, is built around a very specific bet: if a model can verify code, then it should not need humans to hand it a fixed curriculum of reasoning problems. Instead, the model can use the same grounded environment that checks answers to also generate the next training tasks. That turns post-training into a closed loop. The model proposes problems, the executor validates them, the model solves them, and the whole system learns from both sides of the interaction.

That is the part to keep in mind. AZR is not just “RL on code tasks.” It is a self-play recipe where one policy plays two roles at once, the question writer and the question solver. The proposer role tries to invent tasks that are neither trivial nor impossible. The solver role tries to answer the validated task. Both roles update the same underlying model.

What to notice here is the shape of the loop. The proposer does not create abstract text prompts and hope they are useful. It creates a candidate reasoning task, the environment turns that into a concrete program triplet, and only then does the solver try to answer it. The verifier sits in the middle, so the model is always learning from something the executor can actually check.

Why code is the sandbox

AZR needs an environment that is expressive enough to support open-ended reasoning, but strict enough to verify whether a task is valid. Code is a good fit for that balance.

The paper represents each task as a triplet of the form program, input, output. In plain language:

  • deduction means predict the output from the program and input,
  • abduction means recover a plausible input from the program and output,
  • induction means synthesize a program from examples.

Those three directions are different views of the same underlying object. The model is not just memorizing code snippets. It is learning how to move between a program, the data going into it, and the result coming out of it.

This is why the executor matters so much. If the environment can run the code, then it can also turn a partial proposal into a fully labeled training item. That gives AZR a source of synthetic but verifiable practice problems, which is exactly what a zero-data reasoning loop needs.

The proposer is really a curriculum designer

The proposer does more than invent random tasks. It is conditioned on past self-generated examples, plus a task type, and it is asked to produce something different from the examples it has already seen. That “different from the past” constraint matters because it nudges the model toward diversity instead of repetition.

The paper uses a simple intuition for proposer reward. A task is most valuable when the current solver can sometimes solve it, but not always. If the task is too easy, it teaches nothing new. If it is impossible, it also teaches very little. So the proposer is rewarded for tasks with moderate learnability, estimated by running the solver multiple times and measuring how often it succeeds.

In other words, the proposer is trained to find the sweet spot. It is manufacturing practice problems that sit just beyond the model’s current comfort zone, which is a much better curriculum than a fixed human dataset would give it.

How a proposal becomes a real task

The whole pipeline only works if proposals can be turned into valid, deterministic exercises. AZR therefore filters each proposed code task before it reaches the solver.

The validation step has three parts.

  1. Program integrity. The proposed program is executed in Python. If it raises an error or fails to return anything, it is not a usable task.
  2. Program safety. The system blocks sensitive modules and unsafe patterns, so the executor is not exposed to arbitrary environment access.
  3. Determinism. AZR only keeps programs whose output stays the same across repeated executions. That keeps the verifier simple, because a deterministic program gives the same answer every time.

This design choice is important. It narrows the environment, but it also makes reward assignment trustworthy. If the same input can lead to different outputs, then correctness becomes ambiguous and the RL signal gets noisy. AZR avoids that problem by staying inside the deterministic slice of Python.

After validation, the proposal is converted into a concrete task. For deduction and abduction, that means a valid program, input, output triplet. For induction, the proposer supplies a program, a set of inputs, and a free-form message, and the executor fills in the outputs. The message helps because induction is otherwise underconstrained. Without some textual anchor, many different programs could fit the same examples.

Three ways to reason about the same triplet

AZR’s core idea is easier to remember if you treat the three reasoning modes as complementary directions through the same code object.

Deduction

Deduction is the most direct case. The solver sees a program and an input, then predicts the output.

This is the closest to ordinary symbolic simulation. The model has to step through the code, track the variables, and arrive at the right result. In the paper’s examples, the solver often writes down intermediate states while it works, which is exactly what you would hope to see if the model is really simulating the code instead of guessing the answer.

Abduction

Abduction is the reverse problem. The solver sees the program and the output, then tries to recover an input that would produce that output.

This is trickier than deduction because the program may not be one-to-one. There can be many inputs that lead to the same output. The paper handles that by checking whether the solver’s proposed input reproduces the target output when executed, rather than insisting on an exact match with the original hidden input.

That subtlety is important. It turns abduction into a value-equivalence problem instead of an identity problem. The solver is judged by whether its answer works, not whether it guesses the exact hidden seed.

Induction

Induction is the hardest and most interesting mode. The solver gets a handful of input-output pairs, plus the message, and has to synthesize a program that fits the visible examples and also generalizes to the hidden ones.

This is where AZR tests whether the model can move from examples to a reusable function. The held-out cases are there to stop the solver from cheating with a brittle if-else chain that only memorizes the visible inputs. If the synthesized program passes the hidden checks, it is treated as a good induction solution.

If you want a simple mental picture, deduction asks, “What does this program do here?” Abduction asks, “What input would make this happen?” Induction asks, “What function could explain these examples?” Those are different cognitive moves, but they all live inside the same code sandbox.

How the training loop is kept stable

AZR does not just alternate between proposal and solving in the abstract. It uses buffers, formatting rules, and task-specific baselines to keep the loop from collapsing into noise.

The buffers are the model’s short-term memory of valid triplets. At the start, the system seeds them with a tiny set of valid examples, and the paper shows that even an identity-function triplet is enough to kick the process off. After that, newly validated triplets are added back into the buffers, so the model keeps practicing on its own past output.

Those buffers serve two jobs. First, they provide in-context examples to the proposer, which helps it generate fresh tasks instead of repeating itself. Second, if a batch has too few valid proposals, the system can sample earlier validated tasks to fill the gap. That keeps training batches usable even when proposal quality fluctuates.

The reward shaping is also deliberately simple. The solver gets binary correctness reward, while the proposer gets the learnability score. A format penalty is layered on top so the model does not drift away from the expected response structure. The paper uses the DeepSeek R1 style <think> and <answer> format, which gives the model a consistent container for intermediate reasoning and final output.

The update rule, TRR++, is another stability trick. Instead of using one global baseline for everything, AZR keeps separate normalized advantages for each task-role pair, meaning deduction-proposer, deduction-solver, abduction-proposer, and so on. That matters because those roles naturally live on different reward scales. A single global baseline would blur them together and make the gradient noisier than it needs to be.

So the loop is not just self-play in the casual sense. It is self-play plus validation plus role-aware normalization. Each of those pieces exists because the model is generating both the curriculum and the answers, which makes training more powerful but also easier to destabilize.

Why the method is more than a clever trick

The interesting part of AZR is not that it uses code. Plenty of reasoning systems use code. The novelty is that it removes external task curation from the critical path. The model learns what to practice by proposing tasks that the environment can verify, and it learns how to solve them by training on those self-generated tasks.

That matters because the usual bottleneck in reasoning RL is not only reward quality, it is task supply. Human-curated question distributions are expensive and finite. AZR shifts that burden onto the proposer and the executor. As long as the environment can validate tasks and the tasks remain deterministic, the model can keep finding new practice material for itself.

The paper’s results support that idea. The headline models report strong average performance without curated reasoning data, and the ablations suggest the components are not redundant. Removing induction hurts, removing proposer conditioning hurts, and training only the solver also underperforms the full loop. In other words, the gains come from the interaction between proposing and solving, not from either role alone.

Where the approach is fragile

AZR is powerful, but it is also tightly coupled to its environment.

The biggest limitation is that it relies on a code executor, so the reasoning space is narrower than open-world language reasoning. The setup assumes deterministic programs and relatively simple verification. If the tasks are too easy, too unsafe, or too unconstrained, the loop becomes weaker. The paper also notes that some generated chains of thought can become concerning, which means the system still needs oversight even though it no longer depends on human-labeled tasks.

That is the right way to read the method. AZR is not a proof that any model can learn anything from nothing. It is a proof that when a verifier exists, the model can help generate its own training distribution and improve from it. That is a much stronger claim than ordinary RLVR, but it is still conditional on having a grounded environment.

What to remember

The easiest way to remember Absolute Zero Reasoner is as a closed loop with three ingredients:

  • the model proposes its own code reasoning tasks,
  • the executor validates and labels them,
  • the same model then solves those tasks and learns from both proposal quality and answer correctness.

If you keep one intuition from the paper, keep this one: AZR is not just answering questions, it is manufacturing the next questions, but only when the environment can prove they are valid. That is why the method feels like self-play for reasoning, and why the paper treats code execution as the bridge between curriculum generation and verifiable training.
