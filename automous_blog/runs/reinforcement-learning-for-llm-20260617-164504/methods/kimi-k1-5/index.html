Kimi k1.5: Scaling Reinforcement Learning with Long Context

Kimi k1.5 is built around a simple but ambitious idea: if reasoning needs room to think, then reinforcement learning should not force the model to reason in a cramped context window.

That is the paper’s core move. Instead of relying on extra search machinery like Monte Carlo tree search, value networks, or process reward models, it scales the model’s own context window and trains it with a relatively clean RL recipe. The model is allowed to keep exploring, backtracking, and recovering inside one long trajectory, and the training signal only asks one question at the end: did the whole attempt solve the task?

That sounds almost too plain for such a strong system, but the paper’s argument is that this simplicity is the point. Once the model can carry a long reasoning trace, the training loop can reward exploration without needing a separate planner to manage the search.

This overview figure is the right place to start. The training loop alternates between rollout and optimization. Rollout workers generate long reasoning trajectories, the reward signal checks whether the final answer is correct, and the trainer updates the policy before the next round begins. The important thing to notice is that the system is not trying to build a tree of thoughts outside the model. The reasoning happens inside the generated sequence itself.

The basic problem: why ordinary pretraining is not enough

The paper starts from a familiar bottleneck. Next-token pretraining is excellent at scaling compute, but it is still bound by the amount of high-quality text available. At some point, simply predicting the next token on a static corpus stops being the best way to improve reasoning.

Reinforcement learning gives a different axis to scale. Instead of only copying patterns from the dataset, the model can try things, receive reward, and adjust its behavior from experience. For reasoning tasks, that matters because the useful behavior is not just “say the right answer.” It is often “try a line of thought, detect that it is wrong, and recover before the final answer.”

That is why the paper’s design is centered on long chain-of-thought. The long trajectory is not just a verbose explanation. It is the workspace in which the model can plan, reflect, and correct itself.

Start with the right prompts

Before RL can help, the training problems themselves have to be worth learning from. The paper spends real effort on prompt curation because noisy prompts make it easy for RL to learn shortcuts.

The prompt set is supposed to be:

  • diverse, so the model does not only overfit one domain
  • balanced in difficulty, so training does not get stuck on either trivial or impossible tasks
  • objectively verifiable, so reward reflects real success rather than a guess that happens to look plausible

That last point matters a lot. If the task is easy to game, the model can look like it is improving while actually learning to exploit the evaluator. The paper filters out fragile prompt types and tries to remove prompts that can be solved by guessing. In other words, the data is chosen so that the reward means something.

The practical intuition is simple: RL is only as good as the problems you let it practice on. If the task can be hacked with a superficial answer, the model will find that path quickly and stop learning the deeper behavior you wanted.

First teach the model to write long reasoning traces

Before the main RL loop, the model gets a long-CoT supervised warmup.

This is not a big detour from the method. It is the setup that makes the RL phase usable. The warmup teaches the model the style of reasoning the authors want to encourage later: planning before execution, checking intermediate steps, reflecting on mistakes, and exploring alternatives instead of committing too early.

That matters because a raw policy model often does not naturally produce the kind of long, structured reasoning trace that RL needs as training data. The warmup gives the model a head start so that the later RL phase is not trying to bootstrap reasoning from nothing.

Think of it like teaching someone how to keep notes before asking them to solve harder problems. The notes are not the goal. They are the medium that lets the person think more effectively.

The central RL idea: reward the outcome, not a search tree

Here is the cleanest way to understand the method.

Input:

  • a problem x
  • a reference answer y*

The model samples a reasoning trace z and then a final answer y. The reward is based on whether the final answer is correct, using either a rule-based verifier or a trained reward model when exact verification is harder.

So the training objective is just: maximize the expected reward of the model’s generated solution.

That seems almost too simple, but the long context window changes what this objective means in practice. Because the model can generate a long trajectory, the reward no longer says only “did the first attempt work?” It says “did the full attempt eventually solve the task, after trying, revising, and recovering if needed?”

That is the paper’s key conceptual shift. The trajectory itself becomes the reasoning workspace.

Why the policy update is off-policy and mirror-descent-like

The optimization step is where the paper gets more technical, but the intuition is still straightforward.

Instead of updating the policy only from fresh on-policy samples, the method uses an off-policy, relative-entropy-regularized objective. At each iteration, the current model acts as a reference, and the new policy is pushed to improve reward while staying close to that reference.

You can read this as a mirror-descent style step:

  • keep the update anchored to the current policy
  • move toward trajectories that received higher reward
  • do not let the new policy drift wildly in a single jump

That last constraint is important for stability. Long reasoning traces are expensive, so the system wants updates that are strong enough to improve behavior but not so aggressive that they destroy the policy’s structure.

The paper also uses the off-policy setting to reuse data more flexibly. That matters because long trajectories are expensive to generate. If every update required fully fresh rollouts from scratch, the training cost would explode.

Why the paper does not use a value network

This is one of the most interesting design choices in the paper.

Classical RL would often add a value function to estimate which partial reasoning steps are promising. The authors deliberately avoid that. Their reason is subtle but convincing: in long reasoning, a partial path that looks worse right now may still be valuable if it teaches the model how to recover from mistakes.

Suppose a partial chain of thought forks into two next steps. One branch looks cleaner and is closer to the final answer. The other branch makes an error but later recovers and still solves the problem. A value function would tend to favor the cleaner branch and downweight the messy one. But the messy branch may be exactly what the model needs to learn robust exploration.

So the paper chooses a reward signal that judges the final answer and lets the model keep the exploratory path if it eventually reaches a correct solution. That is a deliberate bet: for reasoning, exploration is not just noise. It is part of the skill.

Partial rollouts make long trajectories affordable

If long reasoning is the goal, the obvious problem is cost. Very long trajectories are expensive to generate, and if you discard them after one iteration, you waste a lot of compute.

Partial rollouts are the paper’s practical answer.

This figure shows the mechanism well. Instead of forcing an entire long trajectory to finish in one shot, the system caps the rollout budget. If a trajectory runs long, the unfinished portion is saved and continued in a later iteration. That way, earlier reasoning tokens are not thrown away.

The important intuition is that reasoning is treated like reusable context. Once the model has already written half of a useful attempt, that half can become the starting point for the next step of training or generation. The system then spends new compute only where it still needs to move forward.

This is what makes the long-context recipe scalable. Without partial rollouts, a 128k context window would be impressive but difficult to train efficiently. With partial rollouts, the long trajectory becomes a staged process instead of a single expensive monolith.

The paper also notes that this setup helps with repeated patterns. If the model starts looping or repeating itself, the system can detect that and stop the rollout early. That keeps the model from burning compute on dead ends.

Why the model needs a length penalty

Long reasoning is useful, but there is a trap: the model can start overthinking.

In RL, once the model learns that longer trajectories often correlate with better rewards, it may keep expanding its answers even when the extra tokens are not helping. That is bad for training cost and bad for inference latency.

The paper adds a length penalty to keep that growth under control. The idea is not to punish good reasoning for being detailed. It is to stop the model from treating length itself as a substitute for quality.

The penalty is applied relative to other sampled responses for the same problem. Shorter correct answers are encouraged, while long incorrect answers are pushed down more strongly. That gives the model a reason to be concise when it can be concise, without forcing it to abandon long reasoning when long reasoning is genuinely needed.

This is a good example of the paper’s overall philosophy: do not remove exploration, but do manage its cost.

Sampling strategy is part of the algorithm, not an afterthought

The paper also improves training by choosing which problems to sample.

There are two ideas here. First, curriculum sampling starts with easier tasks and gradually shifts toward harder ones. That makes sense because early in training the model needs a steady stream of solvable tasks. If everything is too hard too early, the model spends compute on failed attempts.

Second, prioritized sampling focuses on problems where the model is currently weak. The model’s own success rate becomes a signal for where to spend more training effort.

This matters because RL efficiency is not just about the optimizer. It is also about whether the batch you choose gives the optimizer something useful to learn. The paper’s ablations show that better sampling beats uniform sampling.

The broader lesson is easy to remember: if the model is already good at a problem, more samples there are less informative than samples from its blind spots.

What the long context window actually buys you

The paper’s scaling story is not just that longer context is “more tokens.” It is that longer context gives the model room to search within a single trajectory.

In this figure, the key thing to notice is the trend rather than any single point. As response length grows, performance tends to improve, especially on harder reasoning tasks. That is strong evidence that the model is using the extra space for useful exploration rather than just padding.

This is an important mental model for the method:

longer context = more opportunities to inspect, revise, and recover before committing to the final answer.

That is the reason the paper can drop heavier search machinery. The reasoning process itself becomes a long internal search trace.

Long-to-short transfer: compress the thinking without losing all the skill

One of the nicest parts of the paper is that it does not stop at long reasoning. It also asks whether that reasoning skill can be compressed into a shorter model for cheaper inference.

The answer is yes, at least partially.

The paper explores several long2short methods, including model merging, shortest rejection sampling, DPO, and a separate long2short RL phase. The goal is to transfer the useful reasoning habits from the long-CoT policy into a short-CoT model that uses fewer test-time tokens.

This comparison figure is useful because it shows the compression story clearly. The short model does not need to reproduce every long trace verbatim. It needs to inherit the reasoning prior: know when to think longer, when to stop, and how to preserve accuracy with less token budget.

Among the options the paper studies, long2short RL is the most direct way to teach that behavior. It keeps the same basic idea as the main method, but now the optimization pressure includes token efficiency more explicitly.

What makes the recipe different from older reasoning systems

The easiest way to place Kimi k1.5 is by contrast.

  • It does not rely on explicit MCTS-style planning trees.
  • It does not need a value network to score every partial path.
  • It does not use a process reward model to supervise every intermediate step.

Instead, it uses a long context window, a final-answer reward, partial reuse of unfinished trajectories, and a policy update that stays close to the current model while still encouraging exploration.

That is a much simpler stack than a search-heavy reasoning system. The tradeoff is that it asks the infrastructure to support long trajectories efficiently. The benefit is that the reasoning policy stays closer to a normal autoregressive model at inference time.

The limits to keep in mind

The paper is strong, but its own limitations are worth remembering.

Long-context training is expensive, so this is not a free recipe. Overthinking still needs explicit control, which is why the length penalty matters. And general verification is still hard for ambiguous tasks, so better verifiers would help.

There is also an open question about how far rollout reuse can scale before the benefit fades. Partial rollouts solve an expensive bottleneck, but they are still a systems compromise, not magic.

What to remember

The main idea in Kimi k1.5 is easier to remember than the engineering around it:

give the model enough context to keep thinking inside one trajectory, reward the final outcome, reuse unfinished rollouts so long reasoning is affordable, and control length so the model does not turn every answer into an essay.

If you keep only one sentence from the paper, make it this:

long-context RL lets reasoning behave like expandable context, so the model can explore, recover, and still be trained from the success of the whole attempt.
