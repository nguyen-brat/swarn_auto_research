Self-Distillation Policy Optimization for Reinforcement Learning with Rich Feedback

SDPO starts from a simple complaint about modern RL for LLMs: a lot of training loops only see a scalar success signal, even when the environment knows much more. In code, for example, the model may get a pass or fail, but the real feedback is the runtime error, the failing test, or the judge explanation that says exactly what went wrong.

That missing information is the bottleneck the paper wants to remove. Rather than train a separate reward model or rely on a stronger external teacher, SDPO lets the current model reread its own attempt with the feedback attached, act as its own retrospective teacher, and then distill those feedback-aware next-token preferences back into the student.

The important shift is from “did the whole answer work?” to “which next-token choices should have changed once the feedback was known?”

The method in one glance

What to notice here is the information flow. RLVR gives the learner only a binary or scalar verdict. RLRF keeps the feedback channel alive, so the model can see the kind of evidence that humans naturally use when debugging: errors, traces, examples, or judge comments.

SDPO is built for that richer setting. It treats feedback as part of the state, not as an afterthought.

Why scalar rewards are too coarse

If the only signal is “correct” or “incorrect,” then all the tokens in a bad solution get pushed around by the same coarse verdict. That makes credit assignment hard.

Suppose a code answer fails because one variable name is wrong near the end. A scalar reward says the whole trajectory was bad, but it does not tell you where the mistake happened. The model has to infer that itself from very little information.

SDPO changes the game by using the feedback to build a better teacher. The teacher sees the original question, the failed attempt, and the feedback, and then it can reinterpret the same prefix with hindsight. That creates a dense logit-level signal instead of a single sequence-level score.

This is why the paper frames the method as “rich feedback” rather than just “better RL.” The extra signal is not a luxury. It is the thing that makes fine-grained credit assignment possible.

The self-teacher loop

This is the whole pipeline in order:

  1. sample a rollout from the current policy,
  2. collect rich feedback from the environment,
  3. reprompt the same model with the feedback attached,
  4. use that feedback-conditioned model as a self-teacher,
  5. distill the teacher’s next-token distribution back into the student,
  6. update the policy on-policy and repeat.

The best way to think about the loop is that the model is not learning from a separate oracle. It is learning from a hindsight version of itself.

That matters because the self-teacher can often do something a scalar reward cannot: it can say not only that the attempt was wrong, but which continuation would have been more plausible given the feedback. In code, that might mean “avoid the bad branch” or “change this return path.” In reasoning tasks, it may mean “don’t take this shortcut, because the feedback now makes that shortcut look inconsistent.”

The paper also allows the feedback to include a successful rollout from the same batch. That is useful when the environment only returns a scalar reward: a good attempt can become a reference point for a failed one. So even in a standard RLVR setting, SDPO can manufacture richer learning signal by comparing failures against successes produced by the current model.

What the loss is doing

The core objective is a token-by-token KL divergence:

LSDPO=∑tKL ⁣(πθ(⋅∣x,y<t)  ∥  stopgrad(πθ(⋅∣x,f,y<t)))\mathcal{L}_{\mathrm{SDPO}}=\sum_t \mathrm{KL}\!\left(\pi_\theta(\cdot\mid x,y_{<t})\;\|\;\mathrm{stopgrad}(\pi_\theta(\cdot\mid x,f,y_{<t}))\right)LSDPO​=t∑​KL(πθ​(⋅∣x,y<t​)∥stopgrad(πθ​(⋅∣x,f,y<t​)))

The student is the model without feedback. The teacher is the same model, but prompted with feedback f. The stopgrad is crucial: it prevents the teacher from collapsing into the student and ignoring the feedback.

Read this objective as “make the student distribution look like the feedback-aware distribution, token by token.”

That is a very different signal from standard RL:

  • RLVR pushes only on the sampled trajectory that happened to be generated.
  • SDPO pushes on the whole next-token distribution at each position.

The paper’s gradient view makes the intuition even clearer. A candidate token gets a positive push if the teacher thinks it is more likely than the student does, and a negative push if the teacher thinks it is less likely. So each position becomes a small local comparison between what the model did and what it would do after rereading the feedback.

That is why the authors describe SDPO as dense credit assignment. The method is not just saying “better answer” or “worse answer.” It is assigning a localized correction signal to many possible next tokens.

Why on-policy matters

SDPO is not off-policy imitation of a frozen teacher. The teacher changes during training, because it is still the same model family evolving over time.

That on-policy choice is doing two jobs:

  • it keeps exploration alive, because the student still samples its own rollouts,
  • it avoids the stale-critic problem that can happen when a fixed teacher is used forever.

The paper reports that the self-teacher itself improves during training, and that the student can eventually surpass the initial teacher. That is the bootstrapping story in practice: the model gets better at retrospective reasoning as it gets better at the task.

There is a catch, though. If the base model is too weak, or if the feedback is noisy or unhelpful, the self-teacher can become unreliable. The paper shows that SDPO can underperform GRPO on weaker models. So this is not a universal replacement for RLVR. It works best when the model already has enough in-context reasoning ability to make sense of the feedback.

The practical tricks that keep it stable

The paper adds a few engineering choices that matter a lot in practice.

First, the teacher is regularized. Instead of letting the feedback-conditioned teacher drift too far, the paper uses either an exponential moving average teacher or a trust-region-style interpolation with the reference teacher. The goal is to keep the teacher responsive to feedback without letting it become unstable.

Second, the distillation loss can be approximated with top-K logits. That is mostly a memory trick: you do not need the full vocabulary distribution to capture the useful part of the signal. The student is usually already concentrated on a small set of next tokens, so the method can focus on those and a tail term instead of storing everything.

Third, the paper notes that the symmetric Jensen-Shannon divergence often stabilizes training better than a plain directional KL. The point is not the exact divergence formula. The point is that this is a self-teaching loop, so the objective needs to be forgiving enough to train without collapsing.

These are all small details, but they are what make the method practical rather than just elegant on paper.

What this buys you in the two main settings

SDPO has two modes of use in the paper.

In standard RLVR environments, it turns successful rollouts into implicit feedback for failed ones. That lets the model learn even when the environment itself is only returning a scalar reward.

In rich-feedback environments, it can use the actual feedback directly. That is where the method is most natural, because the teacher can use runtime errors, judge output, or other tokenized signals to pinpoint what needs to change.

The paper’s empirical story is consistent with that design. On code and reasoning benchmarks, SDPO improves sample efficiency and final accuracy over strong GRPO baselines, and it also helps on very hard test-time discovery tasks where binary reward alone is too sparse to guide learning well.

A useful mental model

The easiest way to remember SDPO is this:

the model rereads its own failed attempt with the failure signal attached, then learns which next-token choices should have been different.

That one sentence captures the whole mechanism.

Unlike ordinary distillation, the teacher is not an external expert. Unlike ordinary RLVR, the signal is not just a scalar verdict. SDPO sits in the middle: it uses the environment to produce feedback, and it uses the model’s own retrospective reasoning to turn that feedback into dense supervision.

What to keep in mind

SDPO is a strong fit when three things are true:

  • the environment can provide informative feedback, not just pass/fail,
  • the base model is strong enough to understand that feedback in context,
  • you want credit assignment that is more precise than a single reward per rollout.

Its main limitations follow directly from those assumptions. Weak models may not benefit as much, noisy feedback can mislead the self-teacher, and the method adds some extra compute for retrospective log-prob evaluation.

So the core lesson is not “self-distillation is always better.” The real lesson is narrower and more useful: when the environment already knows why an attempt failed, let the model learn from that richer signal instead of compressing everything down to one bit.
