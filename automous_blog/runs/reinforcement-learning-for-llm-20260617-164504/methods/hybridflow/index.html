HybridFlow: A Flexible and Efficient RLHF Framework

RLHF looks simple from far away: generate responses, score them, train the actor, repeat. In practice, it becomes a distributed systems problem very quickly. The actor may be large enough to need tensor and pipeline parallelism. The critic and reward model may have different sizes and different memory needs. Generation wants one parallel layout, training wants another, and the data flowing between them has to be reshaped carefully so the GPUs stay busy instead of waiting on transfers.

HybridFlow is built around one idea: treat an RLHF pipeline as a graph of distributed model programs, not as one monolithic trainer. That sounds subtle, but it changes the whole system design. Instead of forcing every model to live inside one giant control loop, HybridFlow separates two levels of work:

  • across models, a single controller coordinates the RLHF dataflow and the transfers between stages;
  • inside each model, a multi-controller execution model handles distributed training, inference, or generation efficiently on the GPUs assigned to that model.

That split is the heart of the paper. It gives HybridFlow two things at once: flexible orchestration across the whole RLHF pipeline, and efficient execution inside each model.

Why RLHF strains existing systems

The pressure points are easy to understand if you picture a PPO-style RLHF iteration.

  1. The actor generates responses from prompts.
  2. The critic, reference policy, and reward model score those responses.
  3. The actor and critic train on the resulting targets and advantages.

Each of those nodes is itself a distributed LLM program. The actor may be training with one parallelism strategy and generating with another. The critic may be trained or evaluated with a different footprint. The reference and reward models may only need forward passes. So the “pipeline” is not a simple sequence of function calls. It is a small DAG whose nodes are full distributed systems.

This is where older designs start to hurt:

  • a pure single-controller design keeps the overall logic clean, but it becomes expensive when it tries to dispatch lots of low-level distributed operations to many workers;
  • a pure multi-controller design keeps GPU-side control local and fast, but it buries data movement and orchestration inside each model’s code, which makes the overall RLHF graph rigid and hard to reuse.

HybridFlow’s answer is to use the right control style at the right level.

The architecture figure is the best way to remember the system. Notice the clean layering: the programming model sits above the distributed engines, and auto-mapping sits beside them as a planner. The paper is really about keeping these concerns separate so they can be optimized independently.

The basic mental model

The simplest way to think about HybridFlow is:

RLHF is a graph of model nodes, each node is a distributed program, and the edges are structured transfers.

That one sentence captures most of the design.

The single controller owns the graph-level view. It knows which model should run next, which outputs need to be transferred, and which stages can overlap because they live on different GPUs. But it does not micromanage every shard of every model.

Each model, by contrast, is wrapped in a worker class. That worker class knows how to do its own local job:

  • actor generation,
  • actor training,
  • critic inference or training,
  • reference-policy forward passes,
  • reward-model scoring.

This is important because different models can reuse existing distributed LLM engines and still fit into one common RLHF orchestration layer.

The hybrid programming model

HybridFlow’s programming model is hierarchical. The hierarchy is not cosmetic; it is what makes the system practical.

At the bottom are worker classes for each model type. These classes encapsulate the model-specific distributed compute. The paper describes base classes for 3D parallelism, ZeRO, and FSDP, so the same RLHF graph can host different execution styles without rewriting the higher-level algorithm.

At the top is the single-controller script that expresses the RLHF algorithm itself. This script reads like the logical algorithm, not like a networking program. For PPO, for example, the code is just a short sequence of primitive calls: generate, compute values, compute rewards, compute advantages, update actor, update critic.

The key detail is that the controller does not need to know how the actor was sharded or how the critic is distributed. It only needs to know the inputs and outputs of each model-level operation.

That is the first big win:

  • model computation stays modular;
  • inter-model communication becomes a reusable protocol;
  • algorithm code stays short and readable.

How data moves between models

If the worker classes are the “inside” of the system, the transfer protocols are the glue between them.

Different models may use different parallelism strategies, so their outputs are not automatically in a format the next model can consume. HybridFlow handles this with transfer protocols built from two pieces:

  • collect, which gathers outputs from the source model’s distributed layout;
  • distribute, which reshapes and sends inputs into the destination model’s layout.

This is a cleaner abstraction than hard-coding send/receive operations in each model. A model operation can be registered with a protocol, and the framework handles the collect/distribute pattern consistently.

The result is that data transfer stops being an ad hoc tangle of point-to-point messages. Instead, it becomes part of the programming model.

That matters because the RLHF graph often involves many-to-many multicast patterns. The actor may produce outputs that feed several downstream consumers. The critic, reference policy, and reward model may each need the same batch in different shard layouts. HybridFlow turns those exchanges into first-class protocols rather than special cases.

Why the paper cares so much about actor training and generation

Among all the nodes in RLHF, the actor is the one that hurts the most.

It appears twice in the pipeline:

  • once for generation, where it autoregressively produces responses;
  • once for training, where it updates its weights.

Those two phases often want different parallelism. Training is usually more compute-heavy and can benefit from larger model parallelism. Generation is more memory-sensitive and may benefit from a different split, often with more data parallelism to keep devices utilized.

If you force both phases to share the same sharding layout, one of them pays the price:

  • generation may underuse GPUs;
  • training may need awkward reshards;
  • or the system may duplicate actor weights to avoid the transition cost.

HybridFlow’s 3D-HybridEngine is the paper’s answer to that transition problem.

3D-HybridEngine: one actor, two layouts

The 3D-HybridEngine is designed to move the actor cleanly between training and generation without wasting memory or spending too much time on resharding.

The idea is straightforward once you strip away the implementation details. Use the same GPU allocation for both phases, but allow the parallel groups to be reorganized between them. That way, you do not need a second copy of the actor, and you do not have to pretend that training and generation have identical workload shapes.

The paper describes the training layout with standard 3D groups, then derives a generation layout that can reuse the same devices while changing the parallel grouping. The subtle part is how the engine chooses the generation groups so that the weights reused from training overlap with the weights needed for generation.

Why does that matter? Because if the group layout is chosen badly, some GPUs end up carrying non-overlapping copies of weights across the transition. That creates redundant memory use and extra communication. HybridFlow’s custom grouping strategy avoids that by ensuring the generation layout is arranged to maximize overlap with the training layout.

In plain language: the engine tries to reshuffle the actor so the same memory can be reused instead of copied twice.

This is where the “zero redundancy” claim comes from. The paper’s point is not just that resharding is possible. It is that resharding can be done in a way that avoids keeping extra actor copies around during the transition.

The practical effect is important for large models. The paper reports that this redesign cuts transition overhead sharply, especially when the actor is large enough that weight movement is no longer a rounding error.

How the actor transition actually flows

It helps to walk through one iteration in the order the system sees it.

  1. The actor finishes training in iteration i.
  2. Its updated parameters are gathered into the layout needed for generation.
  3. Prompts are distributed to the generation replicas.
  4. The actor generates responses.
  5. The generation outputs are gathered and redistributed back into the training layout.
  6. The actor is trained in iteration i + 1.

The important point is that generation and training are not treated as separate models. They are two phases of the same actor, with a controlled transition in between.

HybridFlow gains efficiency because it can reuse the same allocated devices and only move what needs to move, rather than copying a whole duplicate actor just to bridge the phase boundary.

Automatic device mapping

The final piece is placement. Even with a good programming model and a good actor engine, the system still has to decide where each model should live on the cluster.

HybridFlow’s mapping algorithm searches over two things together:

  • how to place the models, either separately or colocated in groups;
  • how to choose a parallelism strategy for each model on the devices it receives.

This is a nice design choice because placement and parallelism are coupled. A model that gets more GPUs may prefer a different parallelism configuration. A colocated set of models may have to share a device pool and execute sequentially. A standalone model may parallelize more aggressively.

The algorithm estimates the cost of each candidate plan with simulators for training, inference, and generation workloads, then picks the plan with the lowest predicted iteration latency.

The intuition is simple:

  • small clusters often benefit from colocating models so the GPUs stay busy;
  • larger clusters often benefit from splitting models so the major stages can run in parallel;
  • the actor often deserves the largest share because generation is hard to parallelize away.

This is not an arbitrary optimizer. It is the system formalizing the same tradeoff a human expert would make by hand, but doing it systematically for the given model sizes and hardware budget.

Why the design is useful in practice

HybridFlow’s real contribution is not just that it runs faster. It is that the speedup comes from architectural clarity.

The framework can express different RLHF algorithms by changing a few lines in the top-level script, while keeping the distributed implementation inside each model reusable. That means PPO, ReMax, and Safe-RLHF can share most of the same machinery even though their dataflows differ.

That reuse is possible because HybridFlow cleanly separates:

  • algorithm logic,
  • model-local distributed execution,
  • inter-model data transfer,
  • cluster placement.

Once those responsibilities are separated, each part can be improved independently. The paper’s evaluation shows that this pays off with large throughput gains over existing systems, especially when the actor transition becomes a bottleneck or when the cluster size changes enough that placement decisions matter.

What to remember

The easiest way to remember HybridFlow is this:

RLHF is not one trainer. It is a graph of distributed model programs.

HybridFlow makes that graph manageable by using:

  • a single controller across model boundaries,
  • multi-controller execution inside each model,
  • reusable transfer protocols for reshaping data between parallel layouts,
  • a zero-redundancy actor engine for training-to-generation transitions,
  • automatic mapping to pick a good placement and parallelism plan.

If you keep that structure in mind, the rest of the paper falls into place. The framework is not trying to simplify RLHF by hiding its distributed nature. It is trying to make that distributed nature explicit, but organized enough that the system remains flexible and fast.
