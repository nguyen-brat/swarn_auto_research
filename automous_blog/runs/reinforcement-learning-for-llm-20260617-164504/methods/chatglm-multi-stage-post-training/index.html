ChatGLM: An Evolving Family of Large Language Models

A pretrained language model can already sound fluent, but that is not the same thing as being a good assistant. It may answer too loosely, miss the user’s intent, behave inconsistently across languages, or fail when the task asks it to choose a tool instead of simply generate text.

ChatGLM’s core lesson is that these problems are not solved by pretraining alone. The paper treats assistant quality as a post-training problem and solves it in stages: first teach the model with supervised examples, then align it with human feedback, and finally specialize part of the family for tool use.

In plain terms, the method is:

pretraining -> supervised fine-tuning -> learning from human feedback -> tool-use alignment -> evaluation

That pipeline is the real subject of the paper. The different GLM-4 models are just the outputs of that pipeline at different scales and with slightly different roles.

What the paper is trying to fix

The starting point is a familiar gap. A base model can be broadly competent and still not feel like an assistant. It may know a lot, but it does not yet know:

  • when to follow an instruction closely,
  • how to stay steady over long context,
  • how to behave well in both Chinese and English,
  • and when to stop generating and reach for a tool.

This is why the paper emphasizes alignment rather than raw capability. The model family is not trying to become better only by being larger. It is trying to become more useful by being shaped after pretraining.

That framing matters because it changes how you read the rest of the paper. The interesting question is not “can the model generate text?” It is “how do we turn a fluent model into a reliable assistant without breaking the abilities it already has?”

The staged recipe

The paper’s post-training recipe is deliberately layered. Each stage does a different job, and each stage protects something the previous one built.

First comes supervised fine-tuning. This is the part where the model sees curated examples of the behavior we want. If the base model is a talented student, supervised tuning is the phase where it is shown what a good answer looks like, not just told to keep predicting the next token.

Then comes learning from human feedback. This stage adjusts the model toward responses people prefer. The important idea is that the model is no longer being optimized only for fluency or imitation. It is being nudged toward the kind of assistant behavior humans actually want in practice.

The paper then extends the aligned family into tool-using variants. That step is not just “add tools.” The model must learn a policy for tool use: when to call a tool, which tool to call, and when to solve the problem directly.

Finally, the family is evaluated as an assistant family, not as a single checkpoint. The paper cares about general benchmarks, instruction following, long-context handling, Chinese alignment, and tool use because those are all different surfaces where post-training can succeed or fail.

Why tool use is a separate stage

Tool use sounds like a simple add-on, but it actually changes the decision-making problem.

Without special alignment, a model may do one of two unhelpful things: it may ignore the tools entirely, or it may overuse them when a direct answer would be easier. The paper’s tool-using extension is meant to teach the model the difference.

That is why the all-tools model is interesting. It is not just a chatbot with access to utilities. It is aligned to understand the user’s intent and decide autonomously whether the job calls for a web browser, Python, text-to-image generation, or a user-defined function.

What to notice in the diagram is the separation between general assistant behavior and tool specialization. The model family does not collapse those concerns into one blob. It first becomes a good assistant, then learns a tool policy on top of that. That ordering is the point. If you reverse it, tool access can easily become noise instead of capability.

Think of the difference like this: a calculator does not make someone mathematically literate. Likewise, giving a model tools does not make it a tool user. The model still has to learn when a tool is the right move.

The flow from input to output

The data flow is easy to remember if you picture the model moving through four successive roles.

It starts as a pretrained multilingual language model, with its strongest footing in Chinese and English. At that point, it knows language, but not yet the assistant contract.

Supervised fine-tuning then anchors the model in explicit examples of desired behavior. This is where the family starts to look like an assistant instead of a generic generator.

Human-feedback alignment pushes the model further toward responses people prefer. This is the stage that tightens the screws on assistant behavior without throwing away the broader competence the model already has.

Tool-use alignment adds the final specialization. Now the model is not only answering. It is also deciding whether the answer should come from its own generation or from a tool-assisted step.

The output is a family of aligned assistants, plus a more specialized all-tools variant that can route tasks through external utilities when needed.

That sequence is the method. The rest of the paper is evidence that this sequence works.

What the results are really saying

The headline results are meant to show that staged post-training does not just improve one narrow benchmark.

GLM-4 is reported to closely rival or outperform GPT-4 on a set of general reasoning and coding benchmarks, including MMLU, GSM8K, MATH, BBH, GPQA, and HumanEval. It also gets close to GPT-4-Turbo on instruction following, matches GPT-4 Turbo (128K) and Claude 3 on long-context tasks, and does better than GPT-4 on Chinese alignment as measured by AlignBench.

The point is not the leaderboard itself. The point is that the same aligned family is being evaluated across several different failure modes:

  • generic reasoning,
  • instruction following,
  • long-context use,
  • Chinese-English behavior,
  • and tool choice.

That spread matters because post-training recipes often look good on one axis while quietly damaging another. The paper’s claim is that this family keeps broad capability while making the assistant behavior more reliable.

What this method assumes

There are a few things this paper does not claim to solve.

It does not expose the training cost or data efficiency of the recipe, so you should not read the abstract as evidence that the pipeline is cheap. It also does not prove that the tool-use recipe will transfer cleanly to every domain or every language mix.

The tool-using behavior likely depends on task-specific supervision and evaluation, which means the “all tools” result is useful but not magically universal. And because the alignment recipe is presented for this model family, portability to other settings is still an open question here.

That is a healthy way to read the work. The method is practical and effective, but it is still a recipe built around a particular family, a particular language mix, and a particular post-training stack.

How to remember the idea

The easiest memory hook is this:

the model becomes an assistant in stages, not in one jump.

First it learns from examples. Then it learns from human preference. Then, once the assistant behavior is stable, it learns a separate policy for tool use. The family design lets those layers build on each other instead of forcing one stage to do everything at once.

So if you want the shortest possible summary, it is this:

ChatGLM shows that assistant quality comes from layered post-training, and that tool use works best when it is added on top of an already aligned model rather than asked to emerge from pretraining alone.
