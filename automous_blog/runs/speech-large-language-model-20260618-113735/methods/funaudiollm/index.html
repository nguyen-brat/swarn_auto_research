FunAudioLLM

FunAudioLLM sits in the speech tokenization, generation, and foundation stacks family, but the main idea is smaller than the name sounds. The paper is building a speech pipeline with three jobs: listen quickly, compress meaningfully, and speak controllably. SenseVoice does the listening, S^3 turns speech into a compact semantic code, and CosyVoice rebuilds speech from that code with prompt-based control.

What to notice in the figure is the division of labor. The system does not force one model to be both an accurate recognizer and a natural generator. It keeps a fast understanding front end, a discrete middle representation, and a generation back end that can reuse the prompt voice and style.

That middle step matters more than it first looks. Once speech has been compressed into the right kind of tokens, the rest of the system becomes much easier to steer. The paper’s real contribution is not just “better ASR” or “better TTS”. It is an interface that lets those two sides talk to each other without losing the cues people actually care about.

SenseVoice: the listening side

SenseVoice is the front end that turns raw waveform into something the rest of the stack can use. It is not only an ASR model. It also handles language identification, speech emotion recognition, and audio event detection, so the system can tell whether the clip is in Chinese or English, whether the speaker sounds happy or sad, and whether the audio contains things like music, laughter, coughing, or applause.

The paper splits this front end into two versions because the usage patterns are different.

SenseVoice-Small is the low-latency version. It is non-autoregressive and encoder-only, which is why it can answer quickly. The input waveform is turned into log-mel features, stacked and downsampled, and then passed through a memory-equipped self-attention encoder. The same encoder can behave like different tasks simply by prepending a small task token: <LID> for language ID, <SER> for emotion, <AEC> for audio event classification, and <ITN/NoITN> for transcript style. That is a compact way to tell one network which job it should do.

The training trick is also worth remembering. For language ID, the model sometimes gets the ground-truth language token during training instead of always having to predict it from scratch. That makes the same setup useful in two modes: the model can either infer the language on its own or be conditioned on a known language token at inference time.

SenseVoice-Large takes the other side of the tradeoff. It is autoregressive, encoder-decoder based, and aimed at broader coverage and higher transcription accuracy. Like Whisper, it uses decoder-side task tokens to specify what should be predicted. The large model can ask for language, emotion, or audio-event outputs, and it can produce timestamps for events when needed. The price is more latency; the reward is support for 50+ languages and better multilingual recognition.

The point of having both models is practical, not redundant. If you need quick response time, use the small model. If you need language breadth and stronger recognition, use the large one. The stack keeps both because voice interaction needs both kinds of behavior.

S^3: a semantic bottleneck that still remembers meaning

Once SenseVoice has understood the signal, the paper does something clever with the intermediate representation. Instead of using a generic speech codec, it builds a supervised semantic tokenizer called S^3.

The important thing to notice is that this tokenizer is not trained like a traditional acoustic codec. It is built from the pre-trained SenseVoice-Large encoder, then a vector quantizer is inserted after the first six layers. The result is a discrete codebook-based bottleneck with a single 4,096-entry dictionary and a token rate of 50 Hz. That means the generator gets one compact semantic token every 20 ms or so instead of a long, noisy acoustic stream.

Why is that helpful? Because older unsupervised tokenizers often preserve sound better than meaning. Residual quantizers and multi-group codecs are good at reconstruction, but their association with text semantics can be weak. That is fine if you only care about sounding like audio. It is not fine if the next model needs tokens that faithfully carry content, language, and some paralinguistic cues.

S^3 is supervised, so it learns the bottleneck in a way that is tied to recognition. The paper trains it end-to-end so the tokens help reduce recognition errors, which gives them a much stronger semantic connection than a purely self-supervised codec would. A small but important detail is the extra positional embedding added after quantization. The codebook tells the model what was said; the position signal helps preserve when it happened.

This is the bridge that makes the rest of FunAudioLLM work. The paper is saying: if you want generation to stay stable, do not feed it a vague acoustic summary. Feed it a compact token stream that already remembers the meaning.

The evidence backs that up. On Common Voice zh-CN, the S^3 tokens beat Whisper-Large-V3 by 4.14% relative error reduction. That does not mean the tokenizer is better at speech recognition than a dedicated recognizer in every setting. It means the discrete tokens are semantically rich enough to survive the round trip through the generator.

CosyVoice: the speaking side

CosyVoice turns the semantic tokens back into speech. It is easier to understand if you think of it as a three-stage speaking machine.

First, an autoregressive Transformer language model generates speech tokens from text plus prompt conditions. Then a flow-matching model reconstructs a mel spectrogram from those tokens. Finally, a HiFTNet-based vocoder turns the mel into a waveform.

That split exists for a reason. The language model is good at sequencing symbolic units. The flow-matching stage is good at recovering acoustic detail. The vocoder is good at turning spectrograms into audible speech. Each stage stays in its lane.

Training follows the same logic. The token LM is trained with teacher forcing, so it learns to predict the next speech token from text and the left-shifted previous tokens. The flow-matching model learns to move from a simple starting distribution toward the target mel spectrogram, conditioned on speech tokens, speaker embedding, and reference speech. Because it follows an optimal-transport-style ODE, it only needs a small number of iterations at inference time, usually about five to ten steps. That is what keeps the reconstruction stage efficient.

The paper also uses classifier-free guidance by dropping most of the preceding conditioning for some training examples. The intuition is simple: if the model only ever sees a very strong prompt, it can become too dependent on that prompt. Randomly masking part of the conditioning teaches it to stay usable even when the prompt signal is imperfect.

Zero-shot cloning is mostly a sequencing problem

CosyVoice’s best-known trick is zero-shot in-context learning. In plain language, it can imitate an arbitrary voice from only a few seconds of reference speech, even across languages.

What to notice here is that cloning is not done by fine-tuning a speaker-specific model. The model reuses the prompt speech as a prefix. If the prompt speech and the target text are in the same language, the prompt speech tokens can be treated as already generated context and the model continues from there. If the target language is different, the system strips away the pieces that would drag the wrong language rhythm into the new utterance, so the prompt keeps the voice identity without forcing the source-language prosody onto the result.

The paper also lets the prompt text come from human annotation or from SenseVoice ASR. That is a neat full-stack connection: the understanding model can help prepare the generation prompt. Speaker embedding and the prompt mel spectrogram are then added on top so timbre and recording conditions stay aligned.

This is why the cloning works from such a short reference clip. The model is not trying to infer an entire speaker from scratch. It is reusing the prompt as a conditioning prefix and then generating the rest of the utterance in the requested language and style.

Instruction fine-tuning makes control explicit

CosyVoice-base is the general-purpose version for identity preservation, zero-shot use, and cross-lingual cloning. CosyVoice-instruct adds another layer: natural language instructions for style control. That gives the user a way to ask for a happy voice, a slower voice, a more emotional delivery, laughter, breaths, or emphasis on specific words.

The paper is careful here. It does not claim that the model magically infers the right tone from arbitrary text. It says the tone becomes controllable when you make the control signal explicit. That is a much more realistic claim.

CosyVoice-sft is the deployment-friendly variant, fine-tuned on seven multilingual speakers so it is ready for immediate use. All three versions share the same core architecture, but they are tuned for different levels of control and deployment convenience.

Why the data strategy matters

The stack only works because the training data is carefully prepared.

SenseVoice-Small is trained on about 300,000 hours across five languages: Chinese, Cantonese, English, Japanese, and Korean. SenseVoice-Large adds another 100,000 hours of multilingual data so the coverage widens to 50+ languages. For rich transcription, the paper uses pseudo labels from existing audio-event and emotion models, which gives it very large SER and AED training sets without manually labeling everything by hand.

CosyVoice uses about 170,000 hours across Chinese, English, Cantonese, Japanese, and Korean. The authors run their own speech detection, SNR estimation, diarization, and separation tools first, then generate pseudo text labels with SenseVoice-Large and Paraformer. Force alignment cleans the data further. That matters because synthesis quality is only as good as the text and timing labels that feed it.

The instruction-tuning data is much smaller, but it is targeted: 101 hours for speaker identity, 407 hours for speaking style, and 48 hours for fine-grained paralinguistics. That is enough to teach the model how to respond to style instructions without overwhelming the base voice model.

What the experiments actually teach

The numbers in the paper are best read as evidence for the pipeline, not as a leaderboard victory lap.

  • SenseVoice-Small transcribes 10 seconds of audio in about 70 ms and is more than 5x faster than Whisper-Small.
  • SenseVoice-Large supports 50+ languages and is stronger than the small model when coverage matters more than latency.
  • S^3 tokens preserve semantics well enough to beat Whisper-Large-V3 on Common Voice zh-CN by 4.14% relative error reduction.
  • CosyVoice on LibriTTS test-clean reaches 2.89% WER and 74.30 speaker similarity, and a 5x ASR re-ranking pass cuts WER to 1.51%.
  • On AISHELL-3, CosyVoice reaches 3.82% CER and 81.58 speaker similarity, with re-ranking lowering CER to 1.84%.
  • CosyVoice-instruct improves emotion control sharply across six emotions compared with CosyVoice-base.

The shape of the results matches the design. SenseVoice buys speed and breadth on the understanding side. S^3 makes the middle representation stable enough to model. CosyVoice then uses that stable bridge to recover voice identity, style, and language-specific speech quality.

The paper also shows that CosyVoice can be used as a data generator. If you synthesize extra ASR data and mix it with the original training set, the downstream recognizer improves, and using more varied text can help more than just duplicating the same text again. That is a useful reminder that the generator is not only an endpoint. It can also become a way to create better training data for other speech systems.

Where the stack still falls short

The paper is practical, but it is not pretending the problem is solved.

  • SenseVoice still struggles more on under-resourced languages.
  • SenseVoice is not designed for streaming transcription.
  • CosyVoice only covers a limited language set.
  • CosyVoice needs explicit style instructions; it does not reliably infer the right emotion from text alone.
  • Singing remains weak.
  • The whole pipeline is not trained end-to-end with an LLM, so errors can propagate from one stage to the next.

That last point is the key tradeoff. The staged design keeps each part simple and controllable, but it also means a mistake in recognition can survive into tokenization or generation. The benefit is robustness and modularity. The cost is that the system is only as strong as its weakest stage.

What to remember

FunAudioLLM is easiest to remember as a three-part sentence:

SenseVoice reads the audio, S^3 compresses it into semantic tokens, and CosyVoice speaks it back with prompt and instruction control.

If you remember one deeper lesson, make it this: speech models get much easier to build when you stop treating waveform as the main representation. The paper works because it chooses a middle representation that is compact enough for generation, but still rich enough to carry meaning, language, speaker identity, and style across the stack.
