VITA-1.5

VITA-1.5 is built around a practical answer to a familiar multimodal problem: if you add vision and speech to the same language model too quickly, the modalities can start fighting each other. The paper’s answer is to stage the learning. First, make the model solid at vision-language. Then teach it to understand speech. Only after that, teach it to answer with speech. That ordering is what lets the system become speech-capable without wiping out the visual competence it already had.

That makes VITA-1.5 a representative method in the multimodal hearing and omni assistants family. Its real contribution is not one exotic module. It is a training recipe that keeps the shared model stable while each new modality is introduced.

The architecture is a shared core with separate entrances and exits

The easiest way to read the model is as two input bridges feeding one LLM, plus a dedicated speech output stack on the far side. Vision and audio are handled separately on the way in. Speech is handled separately on the way out.

What to notice here is the shape of the bottleneck. The model does not ask the LLM to directly swallow raw pixels or raw waveforms. Each modality gets its own encoder and adapter first. The adapters are the narrow bridge into the shared language space, which is exactly where the model wants to keep the interface tight.

A compact way to think about the data flow is:

The visual side keeps both images and videos in the same family of representation. Videos are treated as frame sequences rather than a totally different problem, so the model can reuse the same visual bridge instead of building a separate video backbone. That keeps the architecture cleaner and helps control token growth.

The audio side follows the same principle. The speech encoder compresses the acoustic stream before the LLM sees it. The adapter then reshapes those features into something the shared language model can use. The point is not to preserve every waveform detail. The point is to preserve enough speech structure that the language model can answer correctly.

The output side is the real departure from older speech systems. VITA-1.5 does not depend on an external TTS stack after the fact. It learns to predict discrete speech tokens through a codec-based decoder path, then turns those tokens back into waveform audio. That is what makes the system feel end-to-end instead of transcription plus synthesis glued together.

The paper’s real idea is the order of training

Figure 3 makes the curriculum easy to see. The model does not learn all modalities at once. It learns in passes, and each pass has a different job.

The reason this matters is modality conflict. If vision and speech are mixed together too early, the shared representation can drift in the wrong direction. One modality starts to dominate the gradients, and the other one gets weaker. The paper avoids that by making each stage do one narrow thing before moving on to the next.

Stage 1: teach the model to see before you ask it to listen

Stage 1 is all about vision-language alignment. The model starts with caption data because captions are the gentlest way to connect images to language. At this point, only the visual adapter is trainable. Everything else stays frozen.

That is a sensible first move. The adapter is the translator between visual features and LLM space, so it should learn the mapping before the rest of the model is asked to adapt. If the bridge is wrong, there is no point in letting the whole LLM move around yet.

After that comes vision understanding. Now the visual encoder, the adapter, and the LLM are all trainable, and the model sees the full caption set. This is where the system goes from “I can map image features into language-like vectors” to “I can actually describe what I see.” The LLM is no longer just reading a foreign prefix. It is learning to use visual evidence as part of its own reasoning.

The final visual substage turns descriptions into instruction following. Captions are still mixed in, but the key addition is visual QA. That shifts the model from narrative description to question answering. It is one thing to say what is in a picture. It is another thing to answer, “What color is the object on the left?” or “What is happening in this scene?” Stage 1.3 is where the model learns to obey the visual instruction rather than just produce fluent captions.

The result is important because it shows the visual side still works well after the model later learns speech. After Stage 1, image understanding averages 67.1 on the benchmark suite, which keeps VITA-1.5 close to the strongest open-source multimodal models. In other words, the paper does not buy future speech ability by sacrificing the vision side immediately.

Stage 2: teach it to hear speech as speech, not as noise

Once the visual core is stable, the paper adds audio input. The first audio goal is not conversation. It is alignment.

Stage 2.1 uses speech-transcription pairs and trains the speech encoder with a CTC objective so the encoder learns to recover text from audio. CTC is useful here because it lets the encoder learn the transcript target without requiring a perfect frame-by-frame alignment. That may sound like ordinary ASR, but the role here is narrower. The encoder is learning how to turn time-varying speech into a representation that lives in the same neighborhood as language. This is the bridge that lets the LLM treat speech as understandable context.

After the encoder is trained, the speech adapter connects it to the LLM. At this point the model can read speech input through the shared language backbone. The paper also adds special trainable input tokens that tell the LLM it is dealing with audio, not ordinary text. Think of them as a small speech flag at the front of the prompt. That detail matters because it gives the model a clear cue about the modality it is supposed to process instead of forcing it to infer the whole routing problem from raw features alone.

Stage 2.2 turns that alignment into instruction following. The model is trained on speech QA, but the data mix is intentionally messy in a good way. Some prompts arrive as speech, some as text, and some text prompts are converted into speech form with TTS so the model sees both surface forms. A small classification head also helps the LLM distinguish speech inputs from text inputs. Together, those choices stop the model from overfitting to one input style.

This is the stage where the model starts to feel like a real speech-aware assistant rather than a speech recognizer with a chat interface. It can answer questions that arrive through the voice channel, but the visual skills from Stage 1 are still largely intact. The paper emphasizes that the later audio stages do not erase the earlier visual-language behavior.

Stage 3: teach the model to speak back without adding a separate TTS stack

The final stage is where VITA-1.5 becomes an end-to-end speech system. The paper wants the model to produce spoken replies directly, not just text that is later read out by another module. That is a much harder problem than recognition because the system has to learn a representation that can be turned back into sound.

Stage 3.1 trains a codec model on text-speech pairs. The codec turns speech into discrete tokens and back again. In plain language, it creates a compact alphabet for speech. That alphabet is much easier to predict than raw waveforms, and it gives the model a token-based speech target that fits naturally next to text generation.

Stage 3.2 then teaches the actual speech output path. Text is fed through the tokenizer and the LLM embedding layer to get text embeddings. Those embeddings go into a non-autoregressive decoder first, which builds a global semantic sketch of the utterance. Then an autoregressive decoder refines that sketch token by token, filling in the detailed speech sequence. The codec decoder finally turns the predicted speech tokens back into waveform audio.

A good way to remember the two decoders is to think of them as two passes over the same answer. The NAR decoder draws the broad shape of the response. The AR decoder polishes the fine structure and timing. That division is useful because speech generation needs both global meaning and local acoustic detail. One pass alone is usually too coarse, but two passes can separate the planning problem from the rendering problem.

The crucial stability move here is that the LLM is frozen during speech output tuning. That keeps the new generation capability from rewriting the vision-language and audio-understanding behavior that the previous stages already established. In other words, the model learns to speak without forgetting how to look and listen.

That is the heart of the paper. It does not try to make every module learn everything at once. It isolates the new output skill so the shared core stays stable.

What the results say about the recipe

The evaluation tells a consistent story. Stage 1 already gives strong image-language performance, and the score stays near the top of the open-source pack. After the audio stages, the model largely keeps that visual strength instead of collapsing into a speech-only specialist. On the speech side, the audio stage reaches 2.2% CER on Mandarin ASR and 7.2% WER on English ASR, which is strong enough to show the speech bridge is not just decorative.

Video is the more cautious part of the story. The model is competitive with many open-source systems, but it still trails the strongest proprietary video-capable assistants. That is a useful limitation because it tells you where the staged recipe works well and where it still has room to grow.

The bigger takeaway is not a single metric. It is that the model can absorb speech input and speech output without giving up the visual-language behavior learned earlier. That is exactly what modality conflict had made difficult in the first place.

What to remember

VITA-1.5 is easiest to remember as a three-pass curriculum for a multimodal assistant.

First, make the model solid at vision-language. Second, teach it to understand speech while keeping the visual core intact. Third, freeze the language backbone and teach the model to generate speech tokens through a codec-based output stack.

The adapters are the shock absorbers in that story. They keep each new modality from slamming directly into the shared LLM. The staged order is the other half of the trick. It lets the model add speech without turning the whole system into a tug-of-war between modalities.

The paper’s limitations are also clear. It still depends on large speech-transcription and text-speech corpora, and personalization and long-horizon memory are still open problems. But as a recipe for getting from vision-language MLLMs to near real-time vision and speech interaction, the method is clean and memorable: teach it to see, then hear, then speak.
