SALM - Speech Augmented Language Model

SALM sits in the speech-to-text bridges and alignment recipes family, but it pushes that bridge idea in a more promptable direction. The paper asks a practical question: if a frozen instruction-tuned LLM already knows how to follow text prompts, can we make it follow speech prompts too, without retraining the whole language model?

The answer is yes, but only if the speech side does enough translation work first. SALM uses a pretrained speech encoder, a small modality adapter, and LoRA layers around a frozen LLM. Then it trains the system with multitask speech instruction tuning and speech supervised in-context training so the model learns both the task and the prompt style.

What makes this paper interesting is that it is not just about ASR or translation accuracy. It is about turning speech into something the LLM can read like a prompt, and then teaching the model how to obey extra context such as keyword lists.

The core idea: speech should look like a prompt

The easiest way to read SALM is as a pipeline.

What to notice in the figure is the order of the pieces. SALM does not dump raw audio into the LLM and hope the language model figures it out. It first encodes the speech, then reshapes that representation into the LLM’s space, and only then gives the frozen text model the speech prefix plus any task instructions or keyword context.

In plain language, the audio front end turns speech into a long vector sequence, and the adapter turns that sequence into a shorter prompt-like prefix. Once the prefix is in place, the LLM can do what it already knows how to do: read a prompt and continue it.

That is the whole philosophy of the method. Instead of building a brand-new speech decoder, SALM tries to make a speech-conditioned prompt that a text LLM can naturally consume.

Why the adapter matters

The speech encoder is not enough by itself. Speech and text live at very different rates, and the LLM cannot afford to read raw acoustic frames one by one. SALM handles that mismatch with a small modality adapter made from two Conformer layers with 4x subsampling.

That subsampling is doing two jobs at once. First, it shortens the sequence so the LLM does not drown in audio length. Second, it nudges the representation toward a more language-model-friendly pace. In the paper, the resulting speech prompt has a frame shift of 320 ms, which is a good reminder that this is no longer frame-level audio processing. It is a compressed conditioning signal.

The other adaptation choice is equally important: the LLM stays frozen, and SALM adds LoRA layers with a small rank instead. That keeps the language backbone stable while still giving the model room to adjust to speech input. The audio side learns how to land in the LLM’s embedding space, and the LoRA path learns how to use that new input without rewriting the whole model.

You can think of the adapter as the translator between two very different worlds. The speech encoder extracts the content. The adapter makes that content speak the LLM’s native language.

One model, two tasks

SALM is trained as a multitask speech model, not a single-purpose recognizer. The same backbone is instructed to do ASR and AST, with task instructions choosing which behavior to activate.

That matters because it turns the LLM from a single-task decoder into a unified speech assistant. A prompt can tell it to transcribe, translate to German, or translate to Japanese, and the system should respond with the right kind of text output. The paper uses this instruction style to make the model more general and to improve how well the adapter works across tasks.

This is one of the strongest practical advantages of the approach. The model is not locked into one speech head per task. Instead, the task instruction becomes part of the input, so a single model can cover multiple speech-to-text behaviors.

The result is a much cleaner interface: speech in, instruction in, optional keyword context in, text out.

Keyword boosting as in-context learning

Once speech can enter the LLM as a prompt, the paper tries something more ambitious. Can the model be steered by keywords the way a text LLM can be steered by a few examples or a context block?

The prompt pattern is simple but powerful. A list of keywords is inserted as optional text context before the instruction. The model is then expected to prefer those words when they are relevant to the speech.

This is the important shift: keyword boosting is no longer an external decoding graph or a special rescoring trick. It is a prompt. The model is asked to listen to the audio and also pay attention to the words listed in the context.

That is why the authors describe it as in-context learning for speech-to-text. The boosted words are not learned as special output embeddings, and the model does not need a separate context-biasing graph. At inference time, the keywords are just text sitting in the prompt, and the model decides how to use them.

The idea is elegant, but it only works if the model has been trained to treat that kind of prompt as meaningful. Otherwise, the keyword list is just extra noise.

Speech ICT is the missing training step

That missing step is speech supervised in-context training, or Speech ICT.

The training trick is to augment some supervised speech examples with optional text context. The context is built by sampling words from the current utterance and from other utterances in the dataset, then placing those words into the prompt before the instruction. The paper uses this only on a small fraction of examples, and it tunes the positive ratio of true ground-truth words inside the sampled context.

Why does that help? Because the pretrained LLM already knows what text prompts look like, but it does not automatically know how prompt-style context should behave when the conditioning signal is speech. Speech ICT teaches that connection. It shows the model many examples where the keyword list really matters, so the model learns that the context is not decorative.

This is also why the method generalizes better than a hand-built keyword list might suggest. The sampled context does not just rehearse one vocabulary set. It mixes words from the current utterance and from other utterances, which helps the model see the prompt format rather than memorize a single domain. The paper notes that this can extend to unseen words and even unseen domains.

The positive-ratio setting matters too. If the training prompts contain too many ground-truth words, the model becomes too eager and precision falls. If the ratio is too low, the prompt is not strong enough to teach the biasing behavior. In the reported experiments, a low positive ratio around 6 percent works best.

How the pieces fit together

SALM is easiest to remember as a three-part move:

  1. Encode speech into a continuous representation.
  2. Compress and align that representation so it looks like a prompt prefix.
  3. Train the frozen LLM to treat keyword context and task instructions as real control signals.

That sounds simple, but each step protects a different part of the problem. The speech encoder preserves content. The adapter makes the sequence short enough and compatible enough. The multitask instruction tuning teaches the model the ASR and AST tasks. Speech ICT teaches the model to use in-context keyword biasing instead of ignoring it.

This is why SALM is more than a bridge. It is a bridge plus prompt training.

What the results are really showing

The headline result is not just that SALM can do ASR and AST. It is that one model can cover both tasks and still respond to keyword prompts in a zero-shot style at test time.

For the unified ASR+AST setting, the paper shows that the same system can handle both recognition and translation, with competitive translation quality and acceptable ASR behavior. The exact numbers are less important than the pattern: a frozen LLM plus a small speech bridge is enough to support multiple speech-to-text tasks in one model.

The keyword-boosting results make the main claim even clearer. On the GTC-style ASR test set, SALM improves keyword F-score from 0.35 to 0.56 when the prompt includes the right keyword list. On MuST-C EN-DE, keyword boosting improves F-score from 0.20 to 0.26. That is the paper’s core proof that the model can actually use in-context speech prompts rather than merely accept them.

The ablation on Speech ICT is just as important. Without that training step, the model does not follow keyword prompts well. With it, the prompt starts to matter. In other words, the ability is not a free side effect of using an LLM. It has to be taught into the speech interface.

Where SALM still breaks

The paper is open about the tradeoffs.

  • It can hallucinate or stop early on some utterances.
  • Precision drops as the keyword list gets longer.
  • Long-context handling is still weak.

These failures are exactly what you would expect from a speech model that is trying to behave like a promptable language model. The LLM side gives SALM flexibility, but it also brings the usual autoregressive failure modes. The method is good at rare words, compound words, and normalization tricks, but it is not yet a perfect context-biasing machine.

That tension is the real story of the paper. SALM makes speech promptable, but promptability is not free. If the prompt grows too large or the speech side is not well aligned, the model can drift, hallucinate, or miss the right keyword.

What to remember

SALM’s key idea is simple to state and easy to miss in practice: make speech look like a prompt the LLM already knows how to read, then train the system so keyword context actually works in speech space.

The architecture is a frozen instruction-tuned LLM plus a speech encoder, a small adapter, and LoRA. The training recipe is multitask instruction tuning plus Speech ICT. The payoff is one model that can do ASR, AST, and zero-shot keyword boosting without external biasing graphs.

If you remember one sentence, remember this: SALM turns a speech model into a promptable speech model by teaching the LLM how to treat spoken input and keyword context as the same kind of control problem.
