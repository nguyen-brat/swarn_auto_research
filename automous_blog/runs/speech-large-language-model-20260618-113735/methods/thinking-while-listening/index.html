Thinking while Listening: Chain-of-Thought for Speech Large Language Models

Thinking while Listening is trying to solve a very specific voice-agent problem: the model should reason well, but it should not wait for the end of the user’s sentence if it already has enough meaning to begin. The paper’s answer is to keep reasoning in text, keep a streaming transcript aligned with the user’s speech, and decide when to start chain-of-thought from semantic completeness instead of from a brittle fixed word shift. A second stage then trims those early traces so the system keeps the reasoning gain without paying a large latency cost.

That makes the method feel less like a new speech architecture and more like a timing policy for a multi-stream speech LLM. The model is not only learning what to say. It is also learning when it is safe to begin thinking.

Why latency becomes the real problem

Speech LLMs are attractive because they can listen and answer in one loop, but reasoning changes the time budget. A chain-of-thought trace is useful because it gives the model room to work through the problem step by step. The downside is that every extra reasoning token delays the spoken answer.

That delay matters more in a voice setting than in text chat. In text, the user can tolerate a pause while the model thinks. In speech, a long silence feels like the agent has stalled. So the paper is not just asking, “Can a speech model reason better?” It is asking, “Can it reason earlier, while the question is still arriving?”

The important design choice is to keep the reasoning trace in text space rather than forcing the model to vocalize its thoughts. Text is much denser than speech tokens, so the same reasoning can usually be expressed with far fewer tokens. The paper even tests speech-side CoT, but text reasoning is the cleaner fit once latency is part of the objective.

One model, three live streams

The starting point is Moshi, a multi-stream speech model that already keeps separate channels for user audio, system audio, and system text. That separation is the key enabler. The paper uses the system text channel as a “text monologue” where the model can transcribe the user, reason, and then produce the final response text before the spoken answer is synthesized.

What to notice in the figure is that the reasoning trace is not mixed into the audio stream. It lives on the text monologue side, which lets the model think without having to speak every intermediate step. At the same time, the user transcript stays aligned with the audio, so the model can keep updating its interpretation as new words arrive.

That layout is the whole trick. The paper is not removing speech from the loop. It is giving the model a separate internal lane for thought while preserving the timing of the live conversation.

Two details make that lane work:

  • The model marks reasoning boundaries with special <start_cot> and <end_cot> tokens.
  • When the reasoning trace has to overlap with the live transcript, it uses <switch_cot> and <switch_asr> tokens to alternate between thinking and transcript updates without breaking alignment.

The implementation is careful about length matching too. When one stream needs room and another does not, the model pads the text side and inserts silent audio tokens so the streams remain time-aligned. That sounds mundane, but it is what keeps the architecture usable in real time instead of becoming a loose pile of asynchronous outputs.

How the model learns to start reasoning at the right moment

The hardest part is not generating CoT. It is choosing the moment to begin.

The paper points out two cases where early reasoning can work:

  • the question is effectively complete before the last word arrives, so the model can safely start thinking early and ignore the tail;
  • the question is not fully finished, but enough information is already present to begin reasoning, with later words still refining the answer.

That is why a fixed word-count shift is too crude. Some questions need the final keyword before any reasoning makes sense. Other questions become clear several words earlier. A semantic trigger is better than a syntactic one.

The paper’s answer is a question-completeness score, which acts like a progress bar for meaning.

Purpose: estimate when a partial question has become semantically complete enough to start reasoning.

Intuition: compare what the model expects from a partial question with what it expects from the full question. If those distributions are close, the model already understands enough to think.

Symbols: \zeta is the question-completeness score. p is the current word index in the partial question. N is the full question length. \mathbf{X}_N is the distribution over reasoning and answer when the full question is available. \mathbf{X}_p is the corresponding distribution when only the prefix up to word p is available. \mathbf{X}_0 is the no-question baseline. D_{\mathrm{KL}} is Kullback-Leibler divergence.

Read-aloud meaning: the more the partial question behaves like the full question from the model’s perspective, the closer the score gets to one.

Why it matters: early reasoning should be triggered by meaning, not by a guessed number of words. This score turns the “start thinking now” decision into a semantic estimate instead of a heuristic.

The paper uses an external text language model to estimate the partial-question distributions. That makes the score practical, but it also means the trigger is not learned end-to-end inside the speech model itself. The authors also prefer KL divergence over raw entropy or log-probability because incomplete syntax can make those noisier. In practice, the score can wobble a little as fragments of the question arrive, but the overall trend is still upward.

Once the score crosses a conservative threshold, the system picks that point as the inflection location and shifts the chain-of-thought earlier in the training example. In other words, the model is trained on questions where the reasoning begins as soon as the question is semantically ready, not necessarily when the transcript is finished.

What to notice in this curve is that completeness is not just about the last word. Sometimes one keyword carries the whole question; sometimes the main meaning is already there several words before the end. That is exactly the behavior a fixed word shift misses.

In their setup, the transcript itself is also streamed with a small right-shift look-ahead, six tokens in the paper’s experiments, so the model gets a modest preview without making the transcript feel laggy. That keeps the ASR side useful for reasoning while still staying close to live audio.

Why SFT alone is not enough

The completeness trigger lets the paper create early-thinking training examples, but supervised fine-tuning alone does not fully solve the problem. The model can learn to start earlier, yet still struggle to revise its reasoning when later words arrive. It can also produce traces that are simply too long for easy questions.

The second stage fixes that with preference tuning.

The workflow is straightforward:

  1. Force the early <start_cot> point from the completeness rule.
  2. Sample several candidate early traces from the SFT model.
  3. Keep the answer that is both correct and shorter, and reject the one that is weaker or more verbose.
  4. Train the model with Direct Preference Optimization so it moves toward the preferred trace.

What to notice in the figure is that the preference pairs are not arbitrary “good vs bad” examples. They are generated by rejection sampling from early-start runs, so the model is being taught how to recover from the uncertainty created by starting before the question has fully finished.

The paper uses two flavors of preference data:

  • correctness-based pairs, which favor the trace that gets the answer right and keeps adapting as new words arrive;
  • length-based pairs, which favor the shorter correct trace when the answer does not need a long explanation.

That combination is important. Correctness keeps early reasoning from drifting into the wrong answer. Length control keeps the model from narrating itself into unnecessary latency.

The preference stage is also stabilized in a few careful ways. The DPO comparison is computed only on the text monologue stream, not on the streaming ASR tokens, because the transcript side is mostly alignment scaffolding and would blur the difference between winner and loser traces. The paper also adds a next-token loss on the preferred example and normalizes for length, which helps keep training steady when the preferred trace is short.

How listening and reasoning stay interleaved

There is one more engineering problem hidden inside all of this. If the model starts reasoning before the user has finished speaking, the reasoning tokens and the live transcript tokens want to occupy the same text channel.

The paper solves that with explicit switching tokens and blank-space reuse. First, the streaming ASR transcript is written into the text monologue stream as usual. Then the available [PAD] and [EPAD] slots are reused for chain-of-thought tokens. Whenever the model changes mode, it inserts <switch_cot> or <switch_asr> so the decoder knows whether the next text token should belong to the transcript or to the reasoning trace.

At inference time the system also uses a simple forcing trick. It reserves a few [PAD] tokens at the start so the streaming transcript has room to lag the audio a little, and if the model has not emitted <start_cot> by the time the user finishes, that token is forced in so the answer does not stall.

That is the practical reason the method feels like “thinking while listening” rather than just “thinking early.” The model is not waiting for a clean boundary between listening and reasoning. It is learning to alternate between the two in one synchronized stream.

What this buys the system

The point of all this is not just elegance. The method gives the model a better accuracy-latency trade-off.

Text chain-of-thought brings a large reasoning gain over the baseline speech model, but the completeness trigger makes early reasoning controllable instead of brittle. Then preference tuning trims the trace further, so the model keeps most of the accuracy gain while shedding a large chunk of the latency overhead.

The most useful mental model is this:

  • the transcript tells the model what the user is saying,
  • the completeness score tells the model when enough of the question is known,
  • the CoT stream does the actual reasoning,
  • preference tuning teaches the model not to waste time once it has the answer.

That is why the paper is interesting. It does not merely add CoT to a speech model. It turns reasoning into a timed action inside a live conversation.

What to keep in mind

The limitations are as important as the idea itself. The completeness trigger depends on an external text language model, so it is not a fully self-contained speech-system signal. The training data is synthesized from text reasoning corpora, not from a large native spoken-reasoning dataset. And the evaluation is mostly single-turn and benchmark-shaped, so the paper does not settle how well the method transfers to messy open-ended dialogue.

Even with those caveats, the lesson is durable. If a speech agent has to reason in real time, do not force it to wait for the last word by default. Let it listen, estimate when the question is semantically complete, start reasoning in text, and then use preference tuning to remove the extra slack. That is the core of Thinking while Listening.
