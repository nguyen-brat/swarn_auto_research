Simultaneous Agent Text and Speech Prediction

The paper’s long title sounds technical, but the core idea is easy to say out loud: build a voice agent that can keep listening while it is already preparing and speaking its answer. Instead of treating conversation as a strict back-and-forth queue, the model treats the user and the agent as two synchronized streams. One stream is always coming in. The other stream is always going out.

That is the real shift here. A normal speech stack turns the world into a sequence of separate stages: listen, transcribe, reason, generate text, synthesize speech, then speak. This paper tries to remove that hard boundary. It feeds continuous user speech through a streaming encoder, represents agent speech as discrete codec tokens, aligns both sides at the turn level, and trains one LLM to predict the text and speech channels in parallel.

What to notice in the figure is the split between perception and generation. The user side is a causal streaming encoder, so it can process live speech without waiting for the whole utterance to finish. The agent side is a codec-backed output path, so the model does not need a separate TTS stage after it finishes reasoning. The LLM sits in the middle and learns to connect those two rails.

The big idea

If you want a speech assistant to handle interruption, the model cannot behave like a turn-based chatbot with audio attached. It has to understand that the user may cut in halfway through the agent’s reply, and it has to know how to stop speaking cleanly when that happens. That behavior is called barge-in, and it is one of the main reasons duplex systems exist.

This paper’s answer is not a giant control system. It is a representation choice. The user keeps flowing in as continuous speech features. The agent keeps flowing out as discrete speech codes. Then the model learns both channels together, so the text answer and the spoken realization are produced as part of one duplex process instead of two disconnected steps.

You can think of the whole system as a live loop:

That is why the method feels direct. The model is not juggling a hidden transcript pipeline in the middle. It is learning the conversation shape itself.

Turning speech into something the LLM can work with

The first problem is obvious once you say it plainly: raw waveform audio is too dense to predict directly in a low-latency dialogue loop. The paper solves that by using a neural audio codec to compress agent speech into discrete codes.

Formula card: Codec tokenization

  • Purpose: define the discrete speech representation the model predicts for the agent side.

  • Intuition: a waveform is too fine-grained for a conversational LLM, so the codec turns it into compact symbols over time.

  • Formula:

    CT×N=CodecModel(a)\mathbf{C}_{T\times N}=\textit{CodecModel}(\mathbf{a})CT×N​=CodecModel(a)

  • Symbols: \mathbf{a} is the raw audio signal; \mathbf{C}_{T\times N} is the matrix of acoustic codes; T is the downsampled number of time steps; N is the number of codebooks per step; CodecModel is the neural audio codec.

  • Read-aloud meaning: pass the audio through a codec and get a time-by-codebook grid of discrete speech tokens.

  • Why it matters: once speech becomes a compact sequence of symbols, the LLM can predict speech the same way it predicts text, which makes streaming duplex generation practical.

The codec side is deliberately lightweight. The paper uses a very low-bitrate codec with four codebooks, and the codebooks are independent enough that the model can predict them in parallel at each time step. That is a good fit for live conversation: the agent does not need to wait for a long acoustic rollout before it can keep talking.

This separation also makes personalization easier. Because the user path and the agent path are modeled separately, the agent codec can be fine-tuned toward a target voice without forcing the user perception encoder to change at the same time. In other words, the system can learn to understand speech one way and to sound like the target speaker another way.

Why turn-level alignment is enough

The next design choice is more subtle. The paper does not try to align every word in the user stream with every word in the agent stream. It aligns the two channels at the turn level instead, using the start time of each turn.

That choice sounds coarse, but it is exactly what makes the pipeline manageable. A conversation is not a subtitle track. The important unit for duplex behavior is the turn: who is speaking, when they start, when they stop, and whether one side interrupts the other. The paper also reports that word-level alignment did not help in practice, which is a useful reminder that more granular alignment is not automatically better.

Each turn is marked with separate beginning and end tokens for the text and speech channels. The gaps are padded so the two streams stay synchronized, and the speech side is given a tiny delay. That delay has a practical purpose: the model tends to learn the agent text first, so letting the speech channel lag by a token gives it a little more textual context without adding meaningful latency.

The result is a nice compromise. The model keeps enough structure to know what belongs to the current turn, but it avoids the brittleness of word-by-word synchronization.

Teaching the model to handle interruption

The architecture alone does not teach barge-in. The paper has to show the model what interruption looks like in training.

What matters in this figure is the two-stream construction. The user and agent utterances are split into separate tracks. Whenever one side is speaking, the other side is silenced. That simple trick turns ordinary spoken question-answer pairs and multi-turn conversations into duplex examples the model can learn from.

The paper then adds two important forms of stress:

  • multi-turn dialogue, so the model has to remember the thread of a conversation instead of only responding once
  • barge-in examples, where the user cuts off the agent and the rest of the agent turn is padded with silence

That second part is the key to interruption handling. The model does not learn barge-in because a controller tells it what to do at runtime. It learns barge-in because the training data repeatedly shows a live pattern: the user starts talking again, the agent should stop, and the remaining space in the agent stream should go quiet.

There is also a small but important timing buffer before the agent starts speaking. The purpose of that buffer is not to slow the conversation down for its own sake. It is to reduce false agent takeovers while still keeping the response fast enough to feel interactive. The whole data recipe is really about teaching the model the social rhythm of duplex speech.

Why the separate paths matter for personalization

One of the more useful consequences of the design is that user understanding and agent voice modeling do not have to be entangled.

The user stream goes through a pretrained causal speech encoder. The agent stream goes through the codec. Because those paths are different, the codec can be adapted to a specific voice more directly. That is why the paper can personalize the agent voice without turning the entire duplex system into a speech-pretraining project.

This is the part worth remembering: the model is not trying to make one shared speech representation do every job. It is keeping perception and generation separate enough that each side can be optimized for what it actually needs to do.

What the results are really saying

The numbers in the paper support the story rather than replacing it.

On the hard interruption benchmark, the model stops cleanly in most cases, with a reported barge-in success rate of 94.5% and 0.69 s latency, while keeping false alarms at 0.0%. That means the model is not just talking well in a quiet lab setting; it is reacting correctly when the user cuts in.

On reasoning-style speech tasks, the model scores better than Moshi across the reported sets and stays competitive with an idealized cascaded baseline in some cases, though it still trails that baseline on others. So the method is strong, but it is not magical. Some reasoning capacity is still left on the table.

The personalization result is also important. Fine-tuning the low-bitrate codec improves reconstruction and downstream speech-to-speech quality, even on the target voice. That supports the paper’s broader claim that separate user and agent modeling makes codec adaptation easier, not harder.

What to remember

The cleanest way to remember this paper is as a duplex conversation recipe:

  1. Keep the user as a continuous streaming signal.
  2. Keep the agent as discrete codec tokens.
  3. Align the two sides at the turn level, not at word level.
  4. Train one LLM to predict text and speech channels together.
  5. Teach interruption through the data, not through a separate runtime hack.

That design buys the paper three things at once: low-latency interaction, easier codec personalization, and a much simpler path to building a duplex S2S model from an existing LLM backbone. The tradeoff is that the approach still depends on carefully constructed duplex data and silence padding heuristics, and the reasoning side is not yet consistently better than a best-case cascaded system.

So the main lesson is not “speech models should always be duplex.” It is more specific than that: if you want a conversational agent that can listen while it speaks, model the two directions as synchronized streams from the start. Once you do that, barge-in stops being an afterthought and becomes part of the language model’s job.
