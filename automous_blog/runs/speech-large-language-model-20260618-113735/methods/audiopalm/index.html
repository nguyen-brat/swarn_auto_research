AudioPaLM: A Large Language Model for Speech Understanding and Generation

AudioPaLM is the paper that says: stop treating speech as a second-class input. If you can turn speech into discrete tokens, you can extend a text LLM’s vocabulary, fine-tune one decoder on mixed speech-text tasks, and let the same model handle ASR, AST, TTS, and speech-to-speech translation. The key shift is not “add an audio encoder.” It is “make speech part of the same token world the language model already understands.”

That single design choice explains almost everything in the paper. The model inherits language skill from PaLM-2, learns audio through tokenized speech, and uses plain-text task tags to tell the decoder what job to do. The result is a unified system that can read speech, translate it, speak it back, and even preserve the speaker’s voice across languages.

The big idea: speech becomes another vocabulary

The easiest way to understand AudioPaLM is to forget, for a moment, that speech is continuous. The model never sees raw waveforms. First, speech is compressed into a stream of discrete audio tokens. Then those tokens are simply added to the text model’s vocabulary.

That matters because it changes the problem from “how do we build a special speech model?” to “how do we teach a text decoder to also speak audio tokens?”

What to notice is the symmetry. The left side is a pretrained text-only decoder. The right side is the same decoder after its vocabulary has been expanded with audio tokens. Nothing about the core transformer has to become speech-specific. Once the input has been tokenized, the model simply reads and writes sequences.

That is the central intuition of the paper: if speech is discretized well enough, a strong text LLM can be taught to treat speech like just another sequence of symbols.

How speech is turned into tokens

Before the LLM can do anything useful, raw audio has to be compressed into discrete units. AudioPaLM tries several tokenizers, but the recipe is conceptually the same in every case: take a speech encoder, extract embeddings from audio, and quantize those embeddings into a finite token set.

The important part is not the implementation detail, but the role these tokens play. They are the bridge between continuous sound and a decoder-only language model. If the tokens are weak, the whole system suffers. The paper shows that more capable tokenizers, especially the USM-based ones, lead to much better downstream performance.

That is why the paper is careful about tokenization experiments. AudioPaLM is not claiming that any audio discretization will do. It is saying the opposite: the quality of the audio tokenizer is a major bottleneck, and better speech units translate directly into better ASR, AST, and speech generation.

So when you think about the method, keep this in mind:

  • speech is compressed into a token stream,
  • the token stream becomes part of the model vocabulary,
  • the decoder learns over a shared text-and-audio symbol space.

That shared symbol space is what makes the rest of the design possible.

Extending a text LLM instead of rebuilding one

Once speech has been tokenized, the model only needs a very small architectural change: expand the token embedding matrix so it includes rows for the new audio tokens. The text tokens stay where they are. The audio tokens get fresh embeddings. The transformer stack itself stays decoder-only.

This is a subtle but powerful move. AudioPaLM is not an encoder-decoder system with separate language and speech towers. It is one decoder that has been taught a wider vocabulary.

The paper also keeps the input and output embeddings shared, just as in PaLM and PaLM-2. That matters because it forces the model to use one coherent token space for both reading and generation. If the model predicts audio tokens, it is using the same decoder machinery it uses for text.

The authors also found that full-model finetuning works better than freezing most of the backbone. That makes sense once you think about what the model has to learn. The new audio embeddings are not enough on their own; the entire decoder must adapt to treating audio tokens as meaningful context. So the recipe is:

  • start from a strong text checkpoint,
  • add new audio embeddings,
  • finetune the full model on mixed speech-text data.

This is one of the main lessons of the paper. The text model is not just a convenient starting point. It is the source of the linguistic structure that speech tasks can inherit.

A single decoder, but several possible outputs

After tokenization and vocabulary expansion, the decoder can be used for different kinds of outputs.

Sometimes it produces text, as in ASR or AST. Sometimes it produces audio tokens, as in TTS or S2ST. In the speech-generation path, those audio tokens are then decoded back into waveforms by AudioLM-style stages or by SoundStorm.

So the pipeline has three conceptual steps:

  1. convert speech into discrete audio tokens,
  2. let the unified decoder predict text or audio tokens,
  3. reconstruct waveform audio when the output is meant to be spoken.

That last step is easy to miss, but it is important. AudioPaLM itself does not directly emit waveforms. It predicts audio tokens, and a separate acoustic decoder turns those tokens into speech. This keeps the core model focused on the language problem while reusing an established speech-generation back end.

The paper also uses a short voice-conditioning prompt for speech generation. In practice, that is a small spoken sample, about three seconds long, so the translated speech can inherit the original speaker’s voice instead of falling back to a generic narrator.

Task tags are the steering wheel

A model that can handle text and speech still needs to know what the user wants. AudioPaLM solves that with plain-text task tags.

Examples look like this:

  • [ASR French] for speech recognition,
  • [TTS English] for text-to-speech,
  • [S2ST English French] for speech-to-speech translation.

The tags are not special symbols baked into the architecture. They are ordinary text tokens. That design choice keeps the control interface simple and flexible. The model sees a short instruction, then the input modality, then the continuation it should produce.

This is also why the paper can use one model for several jobs. The task tag tells the decoder whether it should transcribe, translate, speak, or generate intermediate text before producing audio.

A nice detail here is that the tags can be written in more human-readable forms without breaking the method. The exact syntax is less important than the fact that the model gets an explicit task signal. That signal is especially useful when the same speech example can support multiple tasks.

Why the paper likes combined tasks

One of the most interesting ideas in AudioPaLM is that the model is sometimes asked to solve a complex job by writing out intermediate steps in a single autoregressive pass.

For example, instead of directly translating speech to speech, the model can be trained to first produce transcript text, then translated text, and only then audio tokens. That looks a bit like a chain-of-thought style decomposition, except it is happening inside the generation stream rather than through separate models.

The important part is that this is still one model and one decoding process. The model can attend to the original input and everything it has already generated. That is different from a cascade of ASR -> MT -> TTS systems, where each stage hands off a potentially lossy intermediate output to the next model.

Why does this help? Because complex tasks often become easier when the model is allowed to build on simpler ones. In AudioPaLM, the model can first anchor itself in the transcript, then translate from that more stable representation, and finally render speech. The paper reports that this combined-task training improves AST quality, even though it can slightly reduce ASR accuracy. That tradeoff is a recurring theme in the ablations: extra capabilities often cost a little bit of specialization.

Training on mixed speech-text data

AudioPaLM is trained on task mixtures rather than one isolated objective. The paper uses two broad mixtures:

  • an AST-focused mixture for models that output text,
  • a larger S2ST mixture for models that also output speech.

The main idea is not the exact dataset list. The important part is that the same underlying data can be reused for multiple jobs. One speech dataset may support ASR, AST, combined ASR+AST training, or even TTS if it comes with the right text side. That multi-task reuse is one of the ways the model learns to connect speech and text instead of treating them as separate worlds.

The ablations show that this matters. Adding ASR to AST training improves AST. Adding S2ST unlocks new speech-generation ability, but it can slightly reduce text-output accuracy because the model now has to spend capacity modeling audio tokens too. In other words, more tasks make the system richer, but they also make optimization harder.

That is not a failure of the method. It is exactly what you would expect when one decoder is asked to learn both language understanding and speech generation.

What the experiments reveal

The results are best read as evidence that the architecture is doing the right kind of sharing.

AudioPaLM 8B reaches 35.4 BLEU on CoVoST2 AST, and the stronger AudioPaLM-2 8B reaches 37.8 BLEU. The S2ST variant reaches 32.5 ASR-BLEU on CVSS and improves voice similarity over Translatotron 2. AudioPaLM-2 also shows zero-shot AST on FLEURS language pairs that were not seen as speech-translation pairs during training.

Those numbers matter because they line up with the method’s core claim: text pretraining transfers into speech. When the base model is stronger at translation, the speech version inherits that ability more or less immediately, as long as the speech tokens and task prompts are set up well.

The ablations tell the same story from a different angle:

  • better audio tokenization gives better results,
  • finetuning a pretrained checkpoint beats training from scratch,
  • combined tasks help AST,
  • adding S2ST brings speech generation but slightly hurts text tasks,
  • SoundStorm is a strong decoder for turning audio tokens back into waveforms,
  • larger models help a lot.

So the paper is not claiming that speech modeling is solved. It is claiming that the right starting point is a strong text decoder plus a good speech tokenizer, not a speech model built in isolation.

Where the method is fragile

AudioPaLM is powerful, but it has real dependencies.

The first is the tokenizer. If the discrete audio units are poor, the whole model loses quality. The second is finetuning strategy. The authors found that they need to finetune the full model rather than freeze most of the backbone. The third is task balance. Adding speech-generation tasks can pull some performance away from ASR and AST.

Those limitations are worth remembering because they tell you what the method is really buying. AudioPaLM works when the audio side is compressed in a way the language model can use, and when the text pretraining is allowed to adapt to that new token world.

What to remember

The simplest mental model for AudioPaLM is this:

speech waveform -> discrete audio tokens -> one shared text/audio decoder -> text or reconstructed speech

The big idea is not just multimodal input. It is a shared vocabulary. Once speech is turned into tokens, a pretrained text LLM can be extended into a unified speech-text model that handles recognition, translation, synthesis, and speech-to-speech translation with the same decoder.

If you want the shortest summary, it is this: AudioPaLM turns speech into another language of tokens, then teaches a strong text model to speak that language.
